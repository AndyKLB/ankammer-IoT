# CHAPITRE 3 — LINUX DE BASE : SURVIVRE DANS UNE VM

## 3.1 Pourquoi Linux ?

Quand tu fais `vagrant ssh loginS`, tu te retrouves dans un terminal Linux. Si tu n'as jamais utilisé Linux, ce chapitre est pour toi.

**Pourquoi Linux et pas Windows ?**
- Presque tous les serveurs du monde tournent sous Linux (>96%)
- Linux est gratuit et open-source
- Linux est léger, stable, et très configurable
- Kubernetes, Docker, K3s... tout est conçu pour Linux en premier
- Windows Server existe mais est rare dans le monde DevOps/Cloud

### L'interface en ligne de commande (CLI)

Quand tu es connecté à la VM, tu vois quelque chose comme :
```
vagrant@loginS:~$
```

Décortiquons ça :
```
vagrant   = nom de l'utilisateur connecté
@         = "at" (chez)
loginS    = nom de la machine (hostname)
:         = séparateur
~         = répertoire courant ("~" = dossier personnel = /home/vagrant)
$         = invite de commande (tu peux taper ici)
           (si tu étais root/admin, ce serait "#" au lieu de "$")
```

## 3.2 Navigation dans le système de fichiers

### Structure des dossiers Linux

Linux organise tout dans une arborescence unique qui commence à `/` (la racine) :

```
/                        ← Racine (root), tout commence ici
├── bin/                 ← Programmes essentiels (ls, cp, mv...)
├── boot/                ← Fichiers de démarrage
├── dev/                 ← Périphériques (disques, USB...)
├── etc/                 ← Fichiers de configuration système
│   └── rancher/k3s/     ← Configuration K3s (kubeconfig, etc.)
├── home/                ← Dossiers personnels des utilisateurs
│   └── vagrant/         ← Dossier personnel de l'utilisateur "vagrant"
│       └── ~            ← "~" est un raccourci pour ce dossier
├── lib/                 ← Bibliothèques partagées
├── proc/                ← Processus en cours (fichiers virtuels)
├── root/                ← Dossier personnel de root (l'administrateur)
├── tmp/                 ← Fichiers temporaires (effacés au redémarrage)
├── usr/                 ← Programmes et données utilisateur
│   ├── bin/             ← La plupart des programmes installés
│   └── local/bin/       ← Programmes installés manuellement
├── var/                 ← Données variables (logs, bases de données...)
│   ├── log/             ← Fichiers de logs
│   └── lib/rancher/k3s/ ← Données K3s (token, etc.)
└── vagrant → /vagrant   ← Lien vers le dossier partagé Vagrant
```

> **Différence majeure avec Windows :** Sous Windows, tout commence par une lettre de lecteur (`C:\`, `D:\`...). Sous Linux, il n'y a qu'une seule arborescence qui commence à `/`. Les disques supplémentaires sont "montés" dans des sous-dossiers.

### Les commandes de navigation

```bash
# ─── pwd : Print Working Directory ───────────────────────────────
# Affiche le dossier où tu es actuellement
pwd
# Résultat exemple : /home/vagrant

# ─── ls : LiSt ───────────────────────────────────────────────────
# Liste les fichiers et dossiers
ls                  # liste simple
ls -l               # liste détaillée (permissions, taille, date)
ls -la              # liste détaillée + fichiers cachés (commencent par .)
ls -lh              # liste détaillée avec tailles lisibles (Ko, Mo, Go)
ls /vagrant         # lister un dossier spécifique sans s'y déplacer

# Exemple de sortie de ls -la :
# drwxr-xr-x  3 vagrant vagrant 4096 Jan 15 10:30 .
# drwxr-xr-x 20 root    root    4096 Jan 15 10:00 ..
# -rw-r--r--  1 vagrant vagrant  220 Jan 15 09:55 .bash_logout
# drwxr-xr-x  2 vagrant vagrant 4096 Jan 15 10:30 scripts
#
# Lecture : [type+permissions] [liens] [proprio] [groupe] [taille] [date] [nom]
# d = dossier, - = fichier, l = lien symbolique
# rwxr-xr-x = permissions (lire/écrire/exécuter pour proprio, lire/exécuter pour groupe et autres)

# ─── cd : Change Directory ────────────────────────────────────────
cd /vagrant          # aller dans /vagrant (chemin absolu)
cd scripts           # aller dans le sous-dossier "scripts" (chemin relatif)
cd ..                # remonter d'un niveau (.. = dossier parent)
cd ../..             # remonter de deux niveaux
cd ~                 # aller dans le dossier personnel
cd -                 # revenir au dossier précédent
```

### Chemins absolus vs chemins relatifs

```
Chemin absolu : commence par /
  Exemple : /home/vagrant/scripts/setup.sh
  → Toujours le même, peu importe où tu es

Chemin relatif : relatif à ton emplacement actuel
  Si tu es dans /home/vagrant :
  → scripts/setup.sh       pointe vers /home/vagrant/scripts/setup.sh
  → ../                    pointe vers /home/
  → ./setup.sh             pointe vers /home/vagrant/setup.sh
```

## 3.3 Manipulation de fichiers et dossiers

```bash
# ─── mkdir : MaKe DIRectory ──────────────────────────────────────
mkdir mondossier              # crée un dossier
mkdir -p a/b/c                # crée a/, puis a/b/, puis a/b/c/ (récursif)
# L'option -p évite l'erreur si des dossiers intermédiaires n'existent pas

# ─── touch ───────────────────────────────────────────────────────
touch monfichier.txt          # crée un fichier vide (ou met à jour sa date)

# ─── cat : conCATenate ───────────────────────────────────────────
cat monfichier.txt            # affiche le contenu d'un fichier
cat /etc/os-release           # voir la version de l'OS
cat /vagrant/scripts/server.sh # voir un script

# ─── nano : éditeur de texte simple ──────────────────────────────
nano monfichier.txt           # ouvre le fichier dans nano pour l'éditer

# Dans nano :
# Ctrl+O = sauvegarder (O = "write Out")
# Ctrl+X = quitter
# Ctrl+W = rechercher
# Ctrl+K = couper une ligne
# Ctrl+U = coller

# ─── cp : CoPy ───────────────────────────────────────────────────
cp source.txt destination.txt          # copier un fichier
cp -r dossier/ nouveau_dossier/        # copier un dossier (récursif)

# ─── mv : MoVe (déplacer / renommer) ─────────────────────────────
mv ancien.txt nouveau.txt              # renommer
mv fichier.txt /autre/dossier/         # déplacer

# ─── rm : ReMove (supprimer) ─────────────────────────────────────
rm fichier.txt                         # supprimer un fichier
rm -rf dossier/                        # supprimer un dossier et tout son contenu
# ⚠️ DANGER : rm -rf / supprime TOUT le système ! Toujours vérifier le chemin

# ─── chmod : CHange MODe (permissions) ───────────────────────────
chmod +x script.sh            # rendre un fichier exécutable
chmod 755 script.sh           # rwxr-xr-x (lisible et exécutable par tous, modifiable par proprio)
chmod 644 fichier.txt         # rw-r--r-- (lisible par tous, modifiable par proprio seulement)
```

## 3.4 Recherche et filtrage

```bash
# ─── grep : rechercher du texte ──────────────────────────────────
grep "mot" fichier.txt              # cherche "mot" dans le fichier
grep -r "mot" /dossier/             # cherche récursivement dans tous les fichiers
grep -i "MOT" fichier.txt           # insensible à la casse
kubectl get pods | grep Running     # filtrer la sortie d'une autre commande

# ─── | (pipe) : rediriger la sortie ──────────────────────────────
# Le pipe | envoie la sortie d'une commande vers l'entrée d'une autre
kubectl get pods | grep -v Running  # tous les pods SAUF ceux en Running
kubectl get pods | wc -l            # compter le nombre de pods

# ─── > et >> : rediriger vers un fichier ─────────────────────────
echo "bonjour" > fichier.txt        # écrit dans le fichier (écrase)
echo "monde"  >> fichier.txt        # ajoute à la fin du fichier

# ─── head / tail : début / fin ───────────────────────────────────
head -20 fichier.txt                # affiche les 20 premières lignes
tail -20 fichier.txt                # affiche les 20 dernières lignes
tail -f /var/log/syslog             # suit le fichier en temps réel (follow)
                                    # très utile pour voir les logs en direct !
```

## 3.5 Gestion des processus et services

```bash
# ─── systemctl : gestionnaire de services ────────────────────────
# systemd est le gestionnaire de démarrage de Linux moderne
# Il gère les "services" (processus qui tournent en arrière-plan)

sudo systemctl status k3s           # voir l'état du service k3s
sudo systemctl start k3s            # démarrer le service
sudo systemctl stop k3s             # arrêter le service
sudo systemctl restart k3s          # redémarrer le service
sudo systemctl enable k3s           # démarrer automatiquement au boot
sudo systemctl disable k3s          # ne plus démarrer automatiquement

# ─── journalctl : voir les logs des services ──────────────────────
sudo journalctl -u k3s              # tous les logs du service k3s
sudo journalctl -u k3s -n 50        # les 50 dernières lignes
sudo journalctl -u k3s -f           # suivre les logs en temps réel
sudo journalctl -u k3s --since "5 minutes ago"  # logs des 5 dernières minutes

# ─── ps : ProcesseS ──────────────────────────────────────────────
ps aux                              # liste tous les processus
ps aux | grep k3s                   # chercher les processus K3s

# ─── kill : tuer un processus ─────────────────────────────────────
kill 1234                           # arrêter proprement le processus PID 1234
kill -9 1234                        # forcer l'arrêt (signal SIGKILL)
```

## 3.6 Réseau depuis le terminal

```bash
# ─── ip : configurer et voir les interfaces réseau ────────────────
ip a                                # voir toutes les interfaces réseau
ip a show eth1                      # voir seulement eth1
# ou sur Alpine Linux :
ifconfig                            # voir les interfaces (plus ancien)
ifconfig eth1                       # voir seulement eth1

# Sortie typique de ip a :
# 1: lo: <LOOPBACK,UP> mtu 65536 ...
#     inet 127.0.0.1/8 scope host lo       ← loopback (localhost)
# 2: eth0: <BROADCAST,MULTICAST,UP> mtu 1500 ...
#     inet 10.0.2.15/24 ...                ← interface NAT (accès internet)
# 3: eth1: <BROADCAST,MULTICAST,UP> mtu 1500 ...
#     inet 192.168.56.110/24 ...           ← notre interface private_network !

# ─── ping : tester la connectivité ───────────────────────────────
ping 192.168.56.111                 # envoyer des paquets ICMP au worker
ping -c 4 192.168.56.111            # envoyer seulement 4 paquets
# Ctrl+C pour arrêter

# ─── curl : faire des requêtes HTTP ──────────────────────────────
curl http://192.168.56.110          # requête GET sur cette URL
curl -H "Host:app1.com" http://192.168.56.110  # avec en-tête Host personnalisé
curl -v http://192.168.56.110       # verbose : voir les en-têtes complets
curl -s http://192.168.56.110       # silencieux : pas de barre de progression
curl -I http://192.168.56.110       # seulement les en-têtes de réponse

# ─── ss / netstat : voir les ports ouverts ───────────────────────
ss -tlnp                            # voir les ports TCP en écoute avec les processus
# t = TCP, l = listening, n = numérique (pas de résolution DNS), p = processus
```

## 3.7 Alpine Linux : particularités

Alpine utilise **apk** (pas apt) comme gestionnaire de paquets :

```bash
# Installer un paquet
sudo apk add curl wget git

# Mettre à jour les paquets
sudo apk update && sudo apk upgrade

# Chercher un paquet
apk search nginx

# Voir les paquets installés
apk list --installed
```

Alpine utilise aussi `/bin/sh` par défaut (pas `/bin/bash`). Les scripts doivent donc commencer par `#!/bin/sh` et éviter les fonctionnalités spécifiques à bash.

# CHAPITRE 4 — KUBERNETES : COURS COMPLET DE THÉORIE

## 4.1 La philosophie de Kubernetes : l'état désiré

### Le modèle déclaratif

Kubernetes fonctionne sur un principe fondamental appelé **modèle déclaratif** (par opposition au modèle impératif).

**Modèle impératif** (ce que tu évites avec K8s) :
```
"Démarre 3 conteneurs nginx"
"Arrête le conteneur #2"
"Lance un nouveau conteneur nginx"
...
→ Tu donnes des ordres précis, étape par étape
→ Si une étape échoue, tout est compromis
```

**Modèle déclaratif** (ce que K8s utilise) :
```
"L'état désiré : 3 conteneurs nginx qui tournent en permanence"
→ Tu décris ce que tu VEUX, pas comment y arriver
→ K8s se charge de faire en sorte que la réalité corresponde à l'état désiré
→ Si un conteneur plante, K8s le redémarre pour revenir à l'état désiré
```

**Analogie du thermostat :**

```
Mode impératif (tu contrôles manuellement le radiateur) :
  - Tu as froid → Tu mets le chauffage à 100%
  - Après 30 min → Tu l'éteins
  - Tu as encore froid → Tu le rallumes à 50%
  - ... (épuisant et inefficace)

Mode déclaratif (thermostat automatique) :
  - Tu dis : "Je veux qu'il fasse 20°C"
  - Le thermostat surveille la température
  - Quand c'est < 20° → Il allume le chauffage
  - Quand c'est 20° → Il éteint le chauffage
  - La réalité converge automatiquement vers l'état désiré
```

Kubernetes est le thermostat de tes applications.

## 4.2 Architecture complète de Kubernetes

### Vue d'ensemble

Un cluster Kubernetes est composé de deux types de nœuds :

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CLUSTER KUBERNETES                            │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │                   CONTROL PLANE (le "cerveau")                │  │
│  │                                                               │  │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐│  │
│  │  │  API Server  │  │  Scheduler   │  │ Controller Manager   ││  │
│  │  │              │  │              │  │                      ││  │
│  │  │ Point d'entrée│  │ Choisit où  │  │ Surveille et corrige ││  │
│  │  │ de toutes les │  │ placer les  │  │ l'état du cluster    ││  │
│  │  │ commandes    │  │ pods        │  │                      ││  │
│  │  └──────────────┘  └──────────────┘  └──────────────────────┘│  │
│  │                                                               │  │
│  │  ┌──────────────────────────────────────────────────────────┐ │  │
│  │  │                        etcd                              │ │  │
│  │  │         Base de données clé-valeur distribuée            │ │  │
│  │  │         Stocke TOUT l'état du cluster                    │ │  │
│  │  └──────────────────────────────────────────────────────────┘ │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                              │                                      │
│         ┌────────────────────┼───────────────────┐                  │
│         │                   │                   │                  │
│  ┌──────▼──────┐    ┌───────▼─────┐    ┌────────▼────┐             │
│  │  WORKER 1   │    │  WORKER 2   │    │  WORKER 3   │             │
│  │             │    │             │    │             │             │
│  │ [kubelet]   │    │ [kubelet]   │    │ [kubelet]   │             │
│  │ [kube-proxy]│    │ [kube-proxy]│    │ [kube-proxy]│             │

│  │ Pod  Pod    │    │ Pod  Pod    │    │ Pod  Pod    │             │

└─────────────────────────────────────────────────────────────────────┘
```

### Les composants du Control Plane en détail

#### 1. API Server (kube-apiserver)

C'est le **seul point d'entrée** pour toutes les interactions avec le cluster. Quand tu tapes `kubectl get pods`, ta commande va à l'API Server. Quand Argo CD veut déployer quelque chose, il passe par l'API Server. Quand un nœud veut s'enregistrer, il passe par l'API Server.

```
KUBECTL ──────────────────────────────────────────┐
ARGO CD ──────────────────────────────────────────┤
DASHBOARD ────────────────────────────────────────┤──→ API SERVER ──→ etcd
AUTRE TOOL ───────────────────────────────────────┤
KUBELET (des workers) ────────────────────────────┘
```

L'API Server :
- Authentifie les requêtes (qui a le droit de faire quoi ?)
- Valide les requêtes (est-ce que c'est un objet YAML valide ?)
- Sauvegarde dans etcd
- Notifie les autres composants des changements

Analogie : **La standardiste de l'entreprise**. Toutes les communications passent par elle. Elle vérifie que tu as le droit de parler à la personne que tu demandes.

#### 2. etcd

C'est la **base de données** du cluster. Elle stocke tout l'état du cluster : quels pods existent, sur quels nœuds, quelle est leur configuration, quels services existent, etc.

etcd utilise l'algorithme Raft pour être distribué et tolérant aux pannes — si tu as 3 instances etcd et qu'une tombe, les 2 autres continuent à fonctionner.

Analogie : **Les archives et registres de l'entreprise**. Tout ce qui se passe est archivé. Si le bâtiment brûle mais que les archives sont protégées, on peut tout reconstituer.

> **Dans K3s**, etcd est remplacé par **SQLite** par défaut (plus simple, moins de ressources). Pour de la production réelle, on utiliserait etcd.

#### 3. Scheduler (kube-scheduler)

Quand un nouveau pod doit être créé, le Scheduler décide **sur quel nœud le placer**. Il analyse :
- Les ressources disponibles sur chaque nœud (CPU, RAM)
- Les contraintes définies (affinités, tolérances...)
- Les politiques de distribution (éviter de mettre tous les pods d'une même app sur le même nœud)

```
Nouveau pod à placer :
  Requiert : 500m CPU, 256Mi RAM

Nœuds disponibles :
  Node 1 : 2000m CPU libre, 1000Mi RAM libre  ← ✅ Peut accueillir
  Node 2 : 300m CPU libre,  2000Mi RAM libre  ← ❌ Pas assez de CPU
  Node 3 : 1500m CPU libre,  500Mi RAM libre  ← ✅ Peut accueillir

Scheduler choisit Node 1 ou Node 3 (selon son algorithme de scoring)
```

Analogie : **Le responsable de planning RH**. Il regarde qui est disponible, qui a les compétences, et assigne les tâches en optimisant les ressources.

#### 4. Controller Manager (kube-controller-manager)

C'est en réalité **plusieurs contrôleurs** regroupés en un seul processus. Chaque contrôleur surveille un type d'objet Kubernetes et s'assure que la réalité correspond à l'état désiré.

Principaux contrôleurs :

| Contrôleur | Responsabilité |
|------------|----------------|
| **ReplicaSet Controller** | Maintient le bon nombre de pods |
| **Deployment Controller** | Gère les rolling updates |
| **Node Controller** | Surveille l'état des nœuds |
| **Service Account Controller** | Gère les comptes de service |
| **Endpoint Controller** | Maintient les listes d'IP:Port des pods |

**Exemple du ReplicaSet Controller :**
```
État désiré : 3 réplicas de nginx

Boucle de contrôle (s'exécute toutes les quelques secondes) :
1. Compte les pods nginx actuels → trouve 2
2. Etat désiré (3) ≠ État réel (2)
3. Crée 1 pod nginx supplémentaire
4. Compte → trouve 3
5. État désiré (3) = État réel (3) → rien à faire

Si un pod plante :
1. Compte → trouve 2
2. Crée 1 pod → trouve 3
→ Le cluster se "auto-répare" !
```

Analogie : **Un chef de service obsessionnel**. Il vérifie en permanence que son équipe a bien 3 personnes. Si quelqu'un s'absente, il recrute immédiatement.

### Les composants des Worker Nodes

#### 1. kubelet

Le kubelet est l'**agent** qui tourne sur chaque nœud worker. C'est lui qui :
- S'enregistre auprès de l'API Server au démarrage
- Reçoit les instructions (quels pods faire tourner)
- Démarre/arrête les conteneurs via le container runtime
- Surveille l'état des conteneurs et le rapporte à l'API Server

```
API Server ──→ kubelet ──→ Container Runtime (containerd) ──→ Conteneurs
                  ↑
              Surveille
              et rapporte
              l'état
```

Analogie : **Le contremaître de chantier**. Il reçoit les plans de l'architecte, dirige les ouvriers, et rend compte de l'avancement.

#### 2. kube-proxy

Le kube-proxy maintient les **règles réseau** sur le nœud. Il configure iptables (le pare-feu Linux) pour implémenter les Services Kubernetes.

Quand tu crées un Service, kube-proxy s'assure que tout le trafic vers la ClusterIP du Service est redirigé vers les bons pods, sur n'importe quel nœud du cluster.

Analogie : **L'opérateur téléphonique**. Il gère les tables de routage des appels.

#### 3. Container Runtime

C'est le logiciel qui **fait réellement tourner les conteneurs**. Kubernetes ne gère pas directement les conteneurs — il délègue au container runtime via une interface standardisée (CRI = Container Runtime Interface).

Runtimes supportés :
- **containerd** : le plus utilisé (extrait de Docker)
- **CRI-O** : conçu spécialement pour Kubernetes
- **Docker** : via un adaptateur (déprécié dans K8s 1.24+)

Dans K3s, containerd est le runtime par défaut.

## 4.3 Les objets Kubernetes : le référentiel complet

Un **objet Kubernetes** est une entité persistante dans le cluster. Il représente l'état désiré d'un aspect du cluster.

Chaque objet a :
- Un `apiVersion` : quelle version de l'API utiliser
- Un `kind` : quel type d'objet
- Des `metadata` : informations sur l'objet (nom, namespace, labels...)
- Un `spec` : la description de l'état désiré

### Pod — L'unité de base

```
QU'EST-CE QUE C'EST :
  La plus petite unité déployable dans Kubernetes.
  Un Pod contient un ou plusieurs conteneurs qui :
  - Partagent le même réseau (même IP, mêmes ports)
  - Partagent le même stockage (volumes)
  - Ont le même cycle de vie (démarrent et s'arrêtent ensemble)

POURQUOI PLUSIEURS CONTENEURS DANS UN POD ?
  Cas d'usage typique : le pattern "sidecar"
  
  Conteneur principal : ton application web (nginx)
  Conteneur sidecar   : collecteur de logs (envoyés vers Elasticsearch)
  
  Ils partagent le même réseau donc peuvent communiquer via localhost.
  Les logs sont écrits dans un volume partagé.

CARACTÉRISTIQUE IMPORTANTE :
  Les pods sont ÉPHÉMÈRES (ils meurent et ne reviennent pas).
  Si un pod est supprimé, il est REMPLACÉ par un NOUVEAU pod
  avec une NOUVELLE IP.
  C'est pour ça qu'on ne crée presque jamais de pods directement :
  on utilise des Deployments qui les gèrent.
```

**Yaml d'un Pod (pour comprendre, rarement utilisé directement) :**
```yaml
apiVersion: v1          # version de l'API pour les Pods
kind: Pod               # type d'objet
metadata:
  name: mon-pod         # nom du pod (doit être unique dans le namespace)
  namespace: default    # dans quel namespace (défaut = "default")
  labels:               # étiquettes pour identifier ce pod
    app: mon-app        # paire clé: valeur arbitraire
    version: "1.0"
spec:                   # spécification de ce qu'on veut
  containers:           # liste des conteneurs (au moins 1)
  - name: nginx         # nom du conteneur dans le pod
    image: nginx:alpine # image Docker à utiliser
    ports:
    - containerPort: 80 # port exposé par ce conteneur
    env:                # variables d'environnement
    - name: ENV_VAR
      value: "ma_valeur"
    resources:          # limites de ressources
      requests:         # minimum garanti
        memory: "64Mi"  # 64 mégaoctets
        cpu: "100m"     # 100 millicores = 0.1 CPU
      limits:           # maximum autorisé
        memory: "128Mi"
        cpu: "500m"
```

### Deployment — La gestion des réplicas

```
QU'EST-CE QUE C'EST :
  Un Deployment décrit l'état désiré pour un ensemble de pods identiques.
  Il crée et gère un ReplicaSet, qui lui-même gère les Pods.

  Deployment ──gère──→ ReplicaSet ──gère──→ Pod
                                        ──gère──→ Pod
                                        ──gère──→ Pod

POURQUOI UTILISER UN DEPLOYMENT PLUTÔT QU'UN POD ?
  - Garantit le nombre de réplicas (si 1 pod plante → 1 nouveau créé)
  - Permet les mises à jour sans interruption (rolling update)
  - Permet le rollback (revenir à une version précédente)
  - Permet de scaler facilement (changer replicas: 3 → replicas: 10)
```

**Yaml d'un Deployment commenté :**
```yaml
apiVersion: apps/v1     # les Deployments sont dans le groupe "apps"
kind: Deployment
metadata:
  name: mon-deployment
  namespace: default
spec:
  # ─── NOMBRE DE RÉPLICAS ───────────────────────────────────────
  replicas: 3           # Je veux 3 pods identiques qui tournent
  
  # ─── SÉLECTEUR ───────────────────────────────────────────────
  # Le Deployment doit savoir quels pods il gère
  # Il les identifie grâce à leurs labels
  selector:
    matchLabels:
      app: mon-app      # Ce deployment gère les pods avec ce label
  
  # ─── MODÈLE DE POD ────────────────────────────────────────────
  # template décrit comment créer les pods
  template:
    metadata:
      labels:
        app: mon-app    # ← DOIT correspondre au selector ci-dessus
    spec:
      containers:
      - name: nginx
        image: nginx:alpine
        ports:
        - containerPort: 80
  
  # ─── STRATÉGIE DE MISE À JOUR ─────────────────────────────────
  strategy:
    type: RollingUpdate           # mise à jour progressive
    rollingUpdate:
      maxUnavailable: 1           # max 1 pod indisponible pendant la MAJ
      maxSurge: 1                 # max 1 pod supplémentaire pendant la MAJ
```

**Comment fonctionne un Rolling Update :**
```
État initial : 3 pods version v1
[pod-v1-1 Running] [pod-v1-2 Running] [pod-v1-3 Running]

Tu changes l'image de v1 à v2 dans le Deployment

Étape 1 :
[pod-v1-1 Running] [pod-v1-2 Running] [pod-v1-3 Running] [pod-v2-1 Pending]
                                                                    ↓ Running

Étape 2 :
[pod-v1-1 Terminating] [pod-v1-2 Running] [pod-v1-3 Running] [pod-v2-1 Running]
         ↓ Gone

Étape 3 :
[pod-v1-2 Running] [pod-v1-3 Running] [pod-v2-1 Running] [pod-v2-2 Pending]
                                                                  ↓ Running

Étape 4 :
[pod-v1-2 Terminating] [pod-v1-3 Running] [pod-v2-1 Running] [pod-v2-2 Running]

... jusqu'à ce que tous les pods soient en v2

État final : 3 pods version v2
[pod-v2-1 Running] [pod-v2-2 Running] [pod-v2-3 Running]

→ À aucun moment le service n'a été interrompu !
  Il y avait toujours au moins 2 pods disponibles.
```

### Service — L'abstraction réseau stable

```
PROBLÈME QUE LE SERVICE RÉSOUT :
  Les pods ont des IPs qui changent à chaque redémarrage.
  Si ton frontend parle directement à l'IP d'un pod backend,
  et que ce pod redémarre, la communication est cassée.

SOLUTION :
  Le Service est une abstraction avec une IP STABLE (ClusterIP).
  Il maintient une liste à jour des pods disponibles (via Endpoints).
  Il distribue le trafic entre les pods (load balancing basique).

Avant Service :
  Frontend ──→ 10.42.0.15 (pod backend) ← IP change si pod redémarre !

Après Service :
  Frontend ──→ 10.96.45.123 (Service - IP STABLE) ──→ 10.42.0.15 (pod 1)
                                                   ──→ 10.42.0.16 (pod 2)
                                                   ──→ 10.42.0.17 (pod 3)
```

**Les 4 types de Services :**

```yaml
# ─── TYPE 1 : ClusterIP (défaut) ─────────────────────────────────
# IP accessible uniquement à l'INTÉRIEUR du cluster
# Usage : communication entre services (frontend → backend)

apiVersion: v1
kind: Service
metadata:
  name: mon-service-interne
spec:
  type: ClusterIP        # c'est le défaut, peut être omis
  selector:
    app: mon-app         # sélectionne les pods avec ce label
  ports:
  - protocol: TCP
    port: 80             # port du Service (ce que les clients utilisent)
    targetPort: 8080     # port du Pod (ce que l'app écoute réellement)
```

```yaml
# ─── TYPE 2 : NodePort ───────────────────────────────────────────
# Expose le Service sur un port de chaque Node (30000-32767)
# Usage : accès depuis l'extérieur du cluster (pour les tests)

spec:
  type: NodePort
  selector:
    app: mon-app
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30080      # accessible via IP-du-node:30080
```

```yaml
# ─── TYPE 3 : LoadBalancer ────────────────────────────────────────
# Crée un load balancer externe (sur AWS, GCP, Azure...)
# Donne une IP publique accessible depuis Internet
# Usage : production sur cloud provider

spec:
  type: LoadBalancer
  selector:
    app: mon-app
  ports:
  - port: 80
    targetPort: 8080
# → Kubernetes contacte AWS/GCP/Azure pour créer un vrai load balancer
# → Une IP publique est assignée automatiquement
```

```yaml
# ─── TYPE 4 : ExternalName ────────────────────────────────────────
# Redirige vers un nom DNS externe
# Usage : intégrer un service externe (ex: base de données managée)

spec:
  type: ExternalName
  externalName: ma-db.exemple.com
# → Quand un pod fait DNS "mon-service", il obtient "ma-db.exemple.com"
```

### Ingress — Le routeur HTTP intelligent

```
PROBLÈME QUE L'INGRESS RÉSOUT :
  Si tu as 10 applications web, créer 10 LoadBalancer Services serait :
  - Très coûteux (chaque LoadBalancer coûte de l'argent sur le cloud)
  - Difficile à gérer (10 IPs différentes)

SOLUTION : L'INGRESS
  Un seul point d'entrée HTTP/HTTPS qui route vers les bons Services
  selon le nom de domaine ou le chemin URL.

Sans Ingress :
  app1.com ──→ LoadBalancer 1 (IP: 1.2.3.4)  ──→ Service app1
  app2.com ──→ LoadBalancer 2 (IP: 5.6.7.8)  ──→ Service app2
  app3.com ──→ LoadBalancer 3 (IP: 9.10.11.12)──→ Service app3
  → 3 IPs, 3 load balancers, coûteux !

Avec Ingress :
  app1.com ─┐
  app2.com ─┼──→ Ingress Controller (1 seule IP) ──→ Service app1
  app3.com ─┘                                    ──→ Service app2
                                                 ──→ Service app3
  → 1 seule IP, 1 load balancer, économique !
```

**Comment l'Ingress fonctionne en pratique :**

L'Ingress est composé de deux parties :
1. **L'objet Ingress** : les règles de routage (ce qu'on écrit en YAML)
2. **L'Ingress Controller** : le logiciel qui lit les règles et les applique (Traefik dans K3s)

```
Requête : GET / HTTP/1.1
          Host: app1.com
          
→ Traefik reçoit la requête sur le port 80
→ Traefik lit ses règles (issues des objets Ingress K8s)
→ Trouve : Host: app1.com → Service app-one
→ Fait une requête vers le Service app-one
→ Le Service app-one redirige vers un des pods app-one
→ Renvoie la réponse au client
```

**Yaml d'un Ingress commenté :**
```yaml
apiVersion: networking.k8s.io/v1  # groupe API pour l'Ingress
kind: Ingress
metadata:
  name: mon-ingress
  namespace: default
  annotations:
    # Les annotations permettent de configurer l'Ingress Controller
    # Cette annotation dit "utilise Traefik comme controller"
    kubernetes.io/ingress.class: "traefik"
spec:
  rules:
  # ─── RÈGLE 1 : Basée sur le Host ─────────────────────────────
  - host: app1.com              # Si l'en-tête HTTP Host == "app1.com"
    http:
      paths:
      - path: /                 # Pour tous les chemins (/, /page, /api...)
        pathType: Prefix        # "Prefix" = correspond à / et tout ce qui suit
        backend:
          service:
            name: app-one       # → Envoyer vers le Service "app-one"
            port:
              number: 80

  # ─── RÈGLE 2 : Basée sur le Host ─────────────────────────────
  - host: app2.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app-two
            port:
              number: 80

  # ─── RÈGLE PAR DÉFAUT : Sans Host ─────────────────────────────
  # Si aucune règle Host ne correspond → cette règle s'applique
  # IMPORTANT : Doit être EN DERNIER dans la liste
  - http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app-three
            port:
              number: 80
```

### Namespace — Les compartiments isolés

```
QU'EST-CE QUE C'EST :
  Un Namespace est un espace de noms virtuel dans le cluster.
  Il permet de partitionner les ressources du cluster.

POURQUOI UTILISER DES NAMESPACES ?
  - Organisation : séparer dev, staging, production
  - Isolation : une équipe ne peut pas accidentellement modifier les ressources d'une autre
  - Quotas : limiter les ressources par namespace
  - Sécurité : contrôle d'accès par namespace (RBAC)

NAMESPACES PAR DÉFAUT :
  default       → namespace utilisé si tu n'en spécifies pas
  kube-system   → composants internes de K8s (API Server, CoreDNS, Traefik...)
  kube-public   → données lisibles par tous (rares)
  kube-node-lease → heartbeats des nœuds

NAMESPACES DU PROJET :
  argocd  → Argo CD et ses composants (Partie 3)
  dev     → Application déployée par Argo CD (Partie 3)
  gitlab  → Instance GitLab (Bonus)
```

### ConfigMap et Secret

```yaml
# ─── SECRET : données sensibles ──────────────────────────────────
# Les Secrets sont encodés en base64 (PAS chiffrés par défaut !)
# base64 est de l'encodage, pas du chiffrement !
# Pour vraiment sécuriser, utilise des solutions comme HashiCorp Vault

apiVersion: v1
kind: Secret
metadata:
  name: mon-secret
  namespace: default
type: Opaque    # type générique (il existe aussi kubernetes.io/tls, etc.)
data:
  # Les valeurs DOIVENT être en base64
  # echo -n "motdepasse" | base64  → bW90ZGVwYXNzZQ==
  password: bW90ZGVwYXNzZQ==
  username: YWRtaW4=                # "admin" en base64
```

**Utiliser ConfigMap et Secret dans un Pod :**
```yaml
spec:
  containers:
  - name: mon-app
    image: mon-app:v1

    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: ma-config

      valueFrom:
        secretKeyRef:
          name: mon-secret

```

### Labels et Sélecteurs : le système de liaison

Les **labels** sont la colle qui relie les objets Kubernetes entre eux.

```
COMMENT ÇA FONCTIONNE :

1. Tu crées un Deployment qui crée des Pods avec le label "app: frontend"
2. Tu crées un Service avec selector: {app: frontend}
3. Kubernetes relie le Service aux Pods grâce au label

C'est comme des étiquettes de couleur sur des cartons :
- Tu mets une étiquette rouge sur des cartons
- Tu dis "livre tous les cartons rouges à cette adresse"
- Sans connaître exactement combien il y a de cartons ni leurs détails
```

```yaml
# Deployment crée des pods avec labels
spec:
  selector:
    matchLabels:
      app: frontend        # le deployment gère les pods avec ces labels
      tier: web
  template:
    metadata:
      labels:
        app: frontend      # ← les pods auront ces labels
        tier: web
        version: "2.0"     # ← labels supplémentaires possibles

---
# Service sélectionne les pods via les labels
spec:
  selector:
    app: frontend          # ← trouve tous les pods avec app=frontend
    tier: web              # ← ET tier=web
  # Si un pod a app=frontend mais tier=database → PAS sélectionné
```

## 4.5 Le YAML Kubernetes : maîtrise complète

### Les règles du YAML

```yaml
# ─── RÈGLE 1 : Utilise des ESPACES, JAMAIS des tabulations ────────
# ✅ CORRECT
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mon-app

# ❌ INCORRECT (tabulation à la place d'espaces)
spec:
	replicas: 3      # ← TABULATION = ERREUR YAML !

# ─── RÈGLE 2 : L'indentation doit être cohérente ──────────────────
# 2 espaces est la convention dans Kubernetes
# Tu peux utiliser 4 espaces, mais sois cohérent dans tout le fichier

# ─── RÈGLE 3 : Les chaînes avec caractères spéciaux ───────────────
# Sans guillemets (si simple)
name: mon-pod

# Avec guillemets simples (si des : ou des # posent problème)
message: 'Hello: World'

# Avec guillemets doubles (si tu veux des séquences d'échappement)
message: "Ligne1\nLigne2"

# ─── RÈGLE 4 : Les nombres ───────────────────────────────────────
replicas: 3               # entier
cpu: "100m"               # les millicores DOIVENT être en chaîne
memory: "512Mi"           # les tailles mémoire aussi
port: 8080                # entier (port)

# ─── RÈGLE 5 : Les listes ─────────────────────────────────────────
ports:
- containerPort: 80       # élément de liste (tiret + espace)
- containerPort: 443

env:
- name: APP_VERSION
  value: "v1"
- name: APP_PORT
  value: "8080"

# ─── RÈGLE 6 : Les valeurs booléennes ────────────────────────────
enabled: true             # true/false, pas True/False ni TRUE/FALSE

# ─── RÈGLE 7 : Texte multiligne ──────────────────────────────────
# | = préserve les sauts de ligne
script: |
  echo "ligne 1"
  echo "ligne 2"
  echo "ligne 3"

# > = replie les sauts de ligne en espaces (une seule longue ligne)
description: >
  Ceci est une description
  très longue qui continue
  sur plusieurs lignes
# → équivalent à "Ceci est une description très longue qui continue sur plusieurs lignes"

# ─── RÈGLE 8 : Documents multiples dans un fichier ───────────────
# --- sépare plusieurs documents YAML dans un seul fichier
# Très utilisé dans Kubernetes pour mettre Deployment + Service dans le même fichier

apiVersion: apps/v1
kind: Deployment
metadata:
  name: mon-app
# ... spec ...
---                       # ← séparateur de document
apiVersion: v1
kind: Service
metadata:
  name: mon-app
# ... spec ...
```

# --dry-run=client : simule sans envoyer au serveur

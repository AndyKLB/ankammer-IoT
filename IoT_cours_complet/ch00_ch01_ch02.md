# CHAPITRE 1 — COMPRENDRE LES BASES ABSOLUES

## 1.1 Qu'est-ce qu'un serveur ?

Avant de parler de Kubernetes ou de Docker, il faut comprendre ce qu'est un **serveur**.

### La définition simple

Un serveur est **un ordinateur qui rend un service à d'autres ordinateurs**.

C'est tout. La différence entre ton PC et un serveur, c'est uniquement l'usage :
- Ton PC rend service à toi (tu l'utilises directement)
- Un serveur rend service à d'autres machines (tu n'y touches pas directement)

### Exemple concret

Quand tu vas sur `google.com` :
1. Ton navigateur envoie un message : "Donne-moi la page d'accueil de Google"
2. Un serveur Google reçoit ce message
3. Il répond avec le code HTML de la page
4. Ton navigateur affiche la page

```
[Ton PC]  ──── "Donne-moi google.com" ────→  [Serveur Google]
[Ton PC]  ←───── "Voici le HTML" ──────────  [Serveur Google]
```

Google a des **dizaines de milliers** de serveurs dans des immenses bâtiments appelés "datacenters". Ces serveurs fonctionnent 24h/24, 7j/7.

### Le problème avec un seul serveur

Imagine que tu crées un site web et que tu le mets sur un seul serveur. Que se passe-t-il si :
- Ton serveur tombe en panne → ton site est inaccessible
- 10 000 personnes visitent en même temps → le serveur est surchargé et plante
- Tu dois mettre à jour l'application → tu dois arrêter le serveur = site inaccessible

**Solution historique** : acheter plus de serveurs et les faire travailler ensemble. Mais gérer des dizaines de serveurs à la main, c'est cauchemardesque.

C'est là qu'entrent en jeu la virtualisation, Docker et Kubernetes.

## 1.2 L'évolution des serveurs : 30 ans d'histoire en 5 minutes

### Époque 1 (1990-2000) : Un serveur = une application

```
┌─────────────────────┐    ┌─────────────────────┐
│   SERVEUR PHYSIQUE  │    │   SERVEUR PHYSIQUE  │
│                     │    │                     │
│  [Site web]         │    │  [Base de données]  │
│                     │    │                     │
│  CPU : 20% utilisé  │    │  CPU : 15% utilisé  │
│  RAM : 30% utilisée │    │  RAM : 20% utilisée │
└─────────────────────┘    └─────────────────────┘
```

**Problème** : Les serveurs sont **utilisés à seulement 20-30%** de leur capacité. C'est un gaspillage énorme. Un serveur coûte cher (électricité, espace, maintenance).

### Époque 2 (2000-2010) : La virtualisation

Idée révolutionnaire : **faire croire à plusieurs systèmes qu'ils ont chacun leur propre serveur**, alors qu'ils partagent le même matériel.

```
┌──────────────────────────────────────────────────────────────┐
│                     SERVEUR PHYSIQUE                          │
│                                                              │
│   ┌─────────────────┐   ┌─────────────────┐   ┌───────────┐ │
│   │ MACHINE VIRTUELLE│   │ MACHINE VIRTUELLE│   │ VM 3      │ │
│   │ VM 1             │   │ VM 2             │   │           │ │
│   │ [Site web]       │   │ [Base données]   │   │ [Email]   │ │
│   └─────────────────┘   └─────────────────┘   └───────────┘ │
│                                                              │
│   [Logiciel de virtualisation : VirtualBox, VMware, KVM...]  │
│   CPU : 70% utilisé    RAM : 75% utilisée                    │
└──────────────────────────────────────────────────────────────┘
```

**Avantage** : on utilise mieux les ressources.
**Problème restant** : Chaque VM est un OS complet (~500 Mo minimum). Démarrage lent (30-60 secondes). Difficile à déplacer d'un serveur à l'autre.

### Époque 3 (2013-présent) : Les conteneurs

Idée : Au lieu d'isoler un OS entier, on isole **juste l'application et ses dépendances directes**, et on partage le noyau du système.

```
┌──────────────────────────────────────────────────────────────┐
│                     SERVEUR PHYSIQUE                          │
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │Conteneur │  │Conteneur │  │Conteneur │  │Conteneur │     │
│  │[Site web]│  │[DB]      │  │[Email]   │  │[API]     │     │
│  │5 Mo      │  │20 Mo     │  │8 Mo      │  │12 Mo     │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              DOCKER ENGINE                           │   │
│  └──────────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │              NOYAU LINUX PARTAGÉ                     │   │
│  └──────────────────────────────────────────────────────┘   │
│   CPU : 85% utilisé    RAM : 80% utilisée                    │
└──────────────────────────────────────────────────────────────┘
```

**Avantages** :
- Léger : quelques Mo au lieu de centaines
- Rapide : démarre en moins d'une seconde
- Portable : fonctionne identiquement sur n'importe quelle machine avec Docker
- Efficace : maximise l'utilisation des ressources

### Époque 4 (2014-présent) : L'orchestration avec Kubernetes

Maintenant qu'on peut lancer des centaines de conteneurs, comment les gérer ?

```
┌─────────────────────────────────────────────────────────────────┐
│                    CLUSTER KUBERNETES                            │
│                                                                 │
│  [Machine 1]  [Machine 2]  [Machine 3]  [Machine 4]             │
│     ╔════╗      ╔════╗      ╔════╗      ╔════╗                  │
│     ║Cont║      ║Cont║      ║Cont║      ║Cont║                  │
│     ╚════╝      ╚════╝      ╚════╝      ╚════╝                  │
│                                                                 │
│  Kubernetes gère automatiquement :                               │
│  ✓ Placement des conteneurs sur les bonnes machines             │
│  ✓ Redémarrage si un conteneur plante                           │
│  ✓ Augmentation du nombre de copies en cas de pic de trafic     │
│  ✓ Mises à jour sans interruption de service                    │
│  ✓ Distribution équilibrée du trafic entre les copies           │
└─────────────────────────────────────────────────────────────────┘
```

## 1.3 La virtualisation expliquée vraiment depuis le début

### Ton ordinateur physique : de quoi est-il fait ?

Tout ordinateur est composé de **matériel** (hardware) :

| Composant | Rôle | Analogie |
|-----------|------|---------|
| **CPU** (processeur) | Effectue les calculs | Le cerveau |
| **RAM** (mémoire vive) | Stocke les données en cours d'utilisation | Le bureau de travail |
| **Disque dur/SSD** | Stocke les données de façon permanente | L'armoire à dossiers |
| **Carte réseau** | Communique avec d'autres machines | Le téléphone |

### Le système d'exploitation (OS)

Au-dessus du matériel, il y a le **système d'exploitation** (Windows, Linux, macOS). Son rôle est d'être l'intermédiaire entre le matériel et les applications.

```
┌──────────────────────────────────────┐
│          TES APPLICATIONS            │
│  (navigateur, jeux, éditeur texte)   │
├──────────────────────────────────────┤
│        SYSTÈME D'EXPLOITATION        │
│     (Windows / Linux / macOS)        │
│                                      │
│  Gère : mémoire, fichiers, réseau,   │
│         processus, sécurité...       │
├──────────────────────────────────────┤
│            MATÉRIEL                  │
│     CPU  RAM  Disque  Réseau         │
└──────────────────────────────────────┘
```

La partie centrale du système d'exploitation s'appelle le **noyau** (kernel). C'est lui qui parle directement au matériel.

### Qu'est-ce qu'une machine virtuelle exactement ?

Une machine virtuelle (VM) est un **logiciel qui simule un ordinateur complet**. Il prend du CPU, de la RAM et du disque de la vraie machine, et les "présente" à un OS invité comme si c'était du vrai matériel.

```
┌───────────────────────────────────────────────────────────────┐
│                  TON VRAI ORDINATEUR                           │
│                                                               │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │                    VIRTUALBOX                           │  │
│  │                                                         │  │
│  │  ┌──────────────────────┐  ┌──────────────────────┐    │  │
│  │  │  MACHINE VIRTUELLE 1 │  │  MACHINE VIRTUELLE 2 │    │  │
│  │  │                      │  │                      │    │  │
│  │  │  OS : Alpine Linux   │  │  OS : Alpine Linux   │    │  │
│  │  │  CPU virtuel : 1     │  │  CPU virtuel : 1     │    │  │
│  │  │  RAM virtuelle: 1 Go │  │  RAM virtuelle: 512Mo│    │  │
│  │  │  IP: 192.168.56.110  │  │  IP: 192.168.56.111  │    │  │
│  │  │                      │  │                      │    │  │
│  │  │  [K3s serveur]       │  │  [K3s agent]         │    │  │
│  │  └──────────────────────┘  └──────────────────────┘    │  │
│  │                                                         │  │
│  └─────────────────────────────────────────────────────────┘  │
│                                                               │
│  OS : Ubuntu (ou macOS / Windows)                             │
│  CPU réel : 8 cœurs    RAM réelle : 16 Go                     │
└───────────────────────────────────────────────────────────────┘
```

**La VM croit être une vraie machine.** Elle ne sait pas qu'elle est simulée. Elle a son propre OS, ses propres fichiers, sa propre adresse IP. On peut s'y connecter via SSH comme si c'était un vrai serveur distant.

### Comment VirtualBox crée-t-il cette illusion ?

VirtualBox joue le rôle d'**hyperviseur** : il intercepte toutes les opérations que la VM essaie de faire sur le matériel, et les traduit en vraies opérations sur ton matériel réel.

```
VM essaie de lire un fichier
    ↓
VirtualBox intercepte
    ↓
VirtualBox lit dans le fichier disque virtuel (.vdi ou .vmdk)
    ↓
Renvoie le résultat à la VM

VM essaie d'envoyer un paquet réseau
    ↓
VirtualBox intercepte
    ↓
VirtualBox envoie le vrai paquet réseau (ou le redirige via NAT)
    ↓
Renvoie la réponse à la VM
```

### Pourquoi les VMs sont lourdes

Chaque VM embarque un **système d'exploitation complet** — c'est-à-dire des centaines de Mo de fichiers système. De plus, le noyau Linux doit rester actif en permanence, même quand la VM ne fait rien.

```
VM Alpine Linux minimale :
├── Noyau Linux         ~10 Mo
├── BusyBox (outils)    ~2 Mo
├── Bibliothèques C     ~5 Mo
├── Gestionnaire paquets ~1 Mo
└── Total système       ~18-30 Mo (Alpine est vraiment légère !)

VM Ubuntu minimale :
├── Noyau Linux         ~50 Mo
├── Bibliothèques système ~300 Mo
├── Outils de base      ~100 Mo
└── Total système       ~500 Mo - 1 Go

Donc notre VM avec Alpine : ~30 Mo pour le système + K3s (~50 Mo) + ton app
```

C'est pour ça qu'on utilise **Alpine Linux** dans ce projet : c'est la distribution la plus légère qui existe, parfaite pour les VMs et les conteneurs.

## 1.4 Les conteneurs : une révolution légère

### L'idée fondamentale

Les conteneurs ne recréent pas un ordinateur entier. Ils utilisent une fonctionnalité du noyau Linux appelée **isolation de processus**.

Imagine que tu as un appartement (ton ordinateur). 
- Avec une VM : tu construis un appartement entier à l'intérieur de l'appartement (murs, plomberie, électricité... tout)
- Avec un conteneur : tu mets juste des cloisons amovibles pour séparer les espaces. Les murs porteurs (le noyau Linux) restent partagés.

```
VIRTUALISATION                           CONTENEURS
══════════════                           ══════════

┌──────────────────────────────┐         ┌──────────────────────────────┐
│ App A                        │         │ App A    │ App B    │ App C   │
├──────────────────────────────┤         │ libs A   │ libs B   │ libs C  │
│ OS A complet                 │         ├──────────┴──────────┴─────────┤
│ (noyau + tout)               │         │      MOTEUR DE CONTENEURS     │
├──────────────────────────────┤         │           (Docker)             │
│ App B                        │         ├───────────────────────────────┤
├──────────────────────────────┤         │      NOYAU LINUX PARTAGÉ      │
│ OS B complet                 │         ├───────────────────────────────┤
├──────────────────────────────┤         │        MATÉRIEL RÉEL          │
│     HYPERVISEUR              │         └──────────────────────────────┘
├──────────────────────────────┤
│    NOYAU LINUX HÔTE          │
├──────────────────────────────┤
│        MATÉRIEL RÉEL         │
└──────────────────────────────┘

Taille: 300 Mo - 1 Go par VM             Taille: 5 - 100 Mo par conteneur
Démarrage: 30 sec - 2 min                Démarrage: < 1 seconde
```

### Les 3 technologies Linux qui rendent les conteneurs possibles

Tu n'as pas besoin de les maîtriser, mais les comprendre aide à démystifier les conteneurs.

**1. Les Namespaces (espaces de noms)**

Un namespace crée une "vue isolée" du système pour un processus. Il y a 7 types de namespaces :

```bash
# Sans namespace : tous les processus voient tout
PID 1   : systemd       (processus init du système)
PID 234 : nginx         (serveur web dans le "conteneur" A)
PID 567 : mysql         (base de données dans le "conteneur" B)
PID 891 : mon_app       (application dans le "conteneur" C)

# Avec namespace PID : chaque conteneur a son propre espace de noms de processus
Conteneur A voit :
PID 1 : nginx     ← nginx croit être le seul processus qui tourne !

Conteneur B voit :
PID 1 : mysql     ← mysql croit être le seul processus qui tourne !
```

Les types de namespaces :
- `pid` : isolation des processus
- `net` : isolation du réseau (chaque conteneur a sa propre interface réseau)
- `mnt` : isolation du système de fichiers
- `user` : isolation des utilisateurs
- `uts` : isolation du nom d'hôte
- `ipc` : isolation de la communication inter-processus
- `cgroup` : isolation des groupes de contrôle

**2. Les Cgroups (Control Groups)**

Les cgroups permettent de **limiter et surveiller** les ressources utilisées par un groupe de processus.

```
Conteneur A : limité à 512 Mo RAM et 1 CPU
Conteneur B : limité à 1 Go RAM et 2 CPUs
Conteneur C : limité à 256 Mo RAM et 0.5 CPU

→ Même si Conteneur A essaie de prendre plus de RAM, le noyau l'en empêche
→ On garantit que les ressources sont équitablement réparties
```

**3. Union File System (couches de fichiers)**

Les images Docker sont construites par **couches superposées**. Chaque instruction du Dockerfile crée une nouvelle couche.

```
Image "mon-app" :
┌──────────────────────────────────┐
│ Couche 4 : mon code (3 Mo)       │ ← uniquement dans mon-app
├──────────────────────────────────┤
│ Couche 3 : Node.js 18 (50 Mo)    │ ← partagée avec toutes les apps Node.js
├──────────────────────────────────┤
│ Couche 2 : Alpine packages (5 Mo)│ ← partagée avec toutes les apps Alpine
├──────────────────────────────────┤
│ Couche 1 : Alpine Linux (7 Mo)   │ ← partagée avec TOUS les conteneurs Alpine
└──────────────────────────────────┘

→ Si 50 applications utilisent Alpine Linux, les 7 Mo de la couche 1
  ne sont stockés QU'UNE SEULE FOIS sur le disque.
→ Économie d'espace énorme !
```

### Comparaison VM vs Conteneur : le tableau complet

| Critère | Machine Virtuelle | Conteneur Docker |
|---------|------------------|-----------------|
| **Taille** | 300 Mo – 2 Go | 5 – 150 Mo |
| **Démarrage** | 30 sec – 3 min | < 1 seconde |
| **Isolation** | Complète (noyau propre) | Partielle (noyau partagé) |
| **Sécurité** | Très forte | Bonne (quelques vecteurs d'attaque) |
| **Portabilité** | Bonne | Excellente |
| **Performance** | 5-10% overhead | < 1% overhead |
| **Densité** | 10-50 VMs/machine | 100-1000 conteneurs/machine |
| **Cas d'usage** | Isolation forte, OS différents | Microservices, CI/CD, déploiements |

## 1.5 Kubernetes : le chef d'orchestre

### Pourquoi Kubernetes existe

Imagine que tu gères une application populaire avec :
- 500 conteneurs répartis sur 20 machines
- Des pics de trafic imprévisibles
- Des mises à jour toutes les semaines
- Une exigence de disponibilité 99.99% (max 52 min/an d'arrêt)

Sans outil dédié, tu devrais :
- Surveiller manuellement si chaque conteneur fonctionne
- Redémarrer manuellement les conteneurs qui plantent
- Décider manuellement sur quelle machine mettre chaque conteneur
- Gérer manuellement les mises à jour une par une
- Distribuer manuellement le trafic

C'est humainement impossible. C'est pour ça que Google a créé Kubernetes en 2014 (basé sur leur outil interne "Borg" utilisé depuis 2004).

### L'analogie de l'entreprise

Pense à Kubernetes comme au **système de management d'une grande entreprise** :

```
TOI (développeur) = Directeur Général
    Tu définis la stratégie : "Je veux 3 copies de cette app qui tournent"

KUBERNETES = Direction Générale
    Traduit ta stratégie en ordres concrets

API SERVER = Standard téléphonique
    Toutes les demandes passent par lui

ETCD = Système d'archivage de l'entreprise
    Stocke TOUT ce qui se passe

SCHEDULER = Responsable des ressources humaines
    Décide qui fait quoi, où

CONTROLLER MANAGER = Chefs de service
    S'assurent que les objectifs sont atteints

NODES (workers) = Employés
    Exécutent réellement le travail

PODS = Équipes de travail
    Groupes d'employés sur une tâche spécifique

KUBELET = Contremaître
    Supervise les équipes sur chaque site

```

### Ce que Kubernetes automatise concrètement

**Scénario 1 : Un conteneur plante**
```
Sans K8s :
  Conteneur plante → Application inaccessible pendant que tu dors
  Toi le lendemain matin : "Pourquoi l'app était down la nuit ?"

Avec K8s :
  Conteneur plante → K8s détecte en < 10 secondes
  K8s lance un nouveau conteneur sur une autre machine
  Utilisateurs ne voient rien (downtime < 30 secondes)
```

**Scénario 2 : Pic de trafic**
```
Sans K8s :
  Trafic x10 → Serveur surchargé → App lente ou inaccessible
  Toi : tu dois manuellement ajouter des serveurs (si tu es disponible)

Avec K8s + Autoscaler :
  Trafic x10 → K8s détecte la charge élevée
  K8s lance automatiquement 9 conteneurs supplémentaires
  Trafic distribué sur 10 conteneurs → Temps de réponse normal
  Trafic revient à la normale → K8s supprime les 9 conteneurs supplémentaires
```

**Scénario 3 : Mise à jour**
```
Sans K8s :
  Pour mettre à jour : arrêter tous les conteneurs, déployer nouvelle version, redémarrer
  Downtime : 2-5 minutes

Avec K8s (Rolling Update) :
  K8s remplace les conteneurs UN PAR UN :
  - Arrête conteneur 1 → Lance nouveau conteneur 1 → Attend qu'il soit ready
  - Arrête conteneur 2 → Lance nouveau conteneur 2 → Attend qu'il soit ready
  - etc.
  Downtime : 0 secondes !
```

### K3s vs Kubernetes standard

**Kubernetes standard (K8s)** est la version complète, conçue pour des clusters de production avec des centaines ou milliers de nœuds.

**K3s** est une version **allégée** créée par Rancher Labs (rachetée par SUSE), conçue pour :
- L'apprentissage et le développement
- Les environnements aux ressources limitées (Raspberry Pi, edge computing)
- Les petits clusters
- Ce projet exactement !

```
K8s standard :
- Installation : complexe (kubeadm, plusieurs étapes)
- RAM minimale : 1-2 Go par nœud
- Taille binaire : ~100 Mo (plusieurs composants)
- Dépendances : etcd séparé, plusieurs processus

K3s :
- Installation : 1 commande curl
- RAM minimale : 512 Mo par nœud
- Taille binaire : ~50 Mo (tout en 1 binaire)
- Base de données : SQLite intégrée (ou etcd optionnel)
- Inclut : Traefik, CoreDNS, metrics-server, local-path-provisioner
```

**Ce que K3s supprime** par rapport à K8s standard (tout ce dont on n'a pas besoin pour apprendre) :
- Pilotes de stockage cloud (AWS EBS, GCP Persistent Disk, Azure Disk)
- Plugins réseau avancés non utilisés
- Fonctionnalités alpha/beta
- Composants cloud-provider spécifiques

## 1.6 Le réseau : comment les machines se parlent

Avant de plonger dans la pratique, comprendre les bases du réseau est indispensable.

### Les adresses IP

Une **adresse IP** est l'identifiant unique d'une machine sur un réseau, comme une adresse postale.

Format IPv4 : `192.168.56.110`
- 4 nombres séparés par des points
- Chaque nombre va de 0 à 255

### Les plages d'adresses IP privées

Certaines plages d'IP sont réservées aux réseaux **privés** (non accessibles depuis internet) :

| Plage | Usage typique |
|-------|---------------|
| `10.0.0.0` → `10.255.255.255` | Grandes entreprises |
| `172.16.0.0` → `172.31.255.255` | Moyennes entreprises |
| `192.168.0.0` → `192.168.255.255` | Réseaux domestiques, VMs |

Pour ce projet, on utilise `192.168.56.110` et `192.168.56.111` — ce sont des adresses privées dans la plage `192.168.x.x`, parfaites pour des VMs locales.

### Les ports

Un port est un **numéro** qui identifie un service spécifique sur une machine.

```
Adresse IP = numéro de l'immeuble
Port = numéro de l'appartement

192.168.56.110:80   → Machine 192.168.56.110, service sur le port 80 (HTTP)
192.168.56.110:443  → Même machine, service sur le port 443 (HTTPS)
192.168.56.110:6443 → Même machine, API Kubernetes
192.168.56.110:22   → Même machine, SSH
```

Ports bien connus :
| Port | Service |
|------|---------|
| 22 | SSH |
| 80 | HTTP |
| 443 | HTTPS |
| 3306 | MySQL |
| 5432 | PostgreSQL |
| 6443 | API Kubernetes |
| 8888 | Application du projet (wil42/playground) |

### SSH : accès à distance sécurisé

**SSH** (Secure Shell) est le protocole qui permet de **contrôler une machine distante via un terminal**, de façon chiffrée et sécurisée.

```bash
# Syntaxe générale
ssh utilisateur@adresse_ip

# Exemple : se connecter à notre VM serveur
ssh vagrant@192.168.56.110
# Ou avec Vagrant, plus simple :
vagrant ssh loginS
```

Vagrant configure automatiquement SSH sans mot de passe grâce à des **clés SSH** (cryptographie asymétrique — pas besoin de comprendre les détails maintenant).

### Le protocole HTTP et l'en-tête Host

**HTTP** (HyperText Transfer Protocol) est le protocole utilisé par les navigateurs web.

Quand tu tapes `http://app1.com` dans un navigateur :
```
1. Ton navigateur résout "app1.com" en adresse IP (via DNS)
2. Il envoie une requête HTTP :

GET / HTTP/1.1
Host: app1.com          ← L'en-tête Host dit QUEL site tu veux
Accept: text/html
...

3. Le serveur (Traefik/Ingress) lit l'en-tête "Host"
4. Il redirige vers le bon service
```

C'est exactement ce mécanisme qu'on utilise dans la **Partie 2** pour router les requêtes vers app1, app2 ou app3.

## 1.7 Git et GitHub : pourquoi on en a besoin

### Git : système de contrôle de versions

**Git** est un logiciel qui **garde l'historique de tous tes fichiers**. C'est comme avoir un historique de modifications ultra-puissant.

**GitHub** est un site web qui héberge des dépôts Git en ligne. C'est comme Google Drive, mais pour du code, avec toutes les fonctionnalités de Git.

Dans ce projet, GitHub sert de **source de vérité** pour la Partie 3 : tu y mets tes fichiers de configuration Kubernetes, et Argo CD les lit pour déployer automatiquement.

```bash
# Synchroniser avec GitHub

# Le workflow typique

# → Argo CD détecte le push et met à jour l'application !
```

# CHAPITRE 2 — VAGRANT : AUTOMATISER LA CRÉATION DE VMs

## 2.1 Qu'est-ce que Vagrant, vraiment ?

**Vagrant** est un outil en ligne de commande qui automatise la création, la configuration et la gestion de machines virtuelles.

### L'analogie du LEGO

Imagine que tu veux construire une maison en LEGO. Tu as deux options :

**Option A (sans Vagrant)** : Tu ouvres VirtualBox, tu cliques partout, tu configures manuellement chaque paramètre, tu télécharges un ISO, tu installes Linux... 2 heures plus tard, tu as ta VM.

**Option B (avec Vagrant)** : Tu écris un "plan" dans un fichier texte (`Vagrantfile`), puis tu tapes `vagrant up`. 10 minutes plus tard (pour le téléchargement initial), tu as ta VM exactement comme tu l'avais décrite.

La deuxième option est reproductible : si quelqu'un d'autre veut la même VM, il prend ton fichier et fait `vagrant up`. Il obtient exactement la même chose.

### Le concept de "box" Vagrant

Une **box** est une image de VM pré-construite, comme un moule. Au lieu de repartir de zéro (installer Linux depuis l'ISO), on part d'une box prête à l'emploi.

```
BOX = Image de VM pré-construite
  ↓ (vagrant up utilise la box comme base)
VM = Instance créée à partir de la box
  ↓ (Vagrant personnalise la VM selon le Vagrantfile)
VM configurée = Ce qu'on utilise réellement
```

Les boxes sont hébergées sur https://app.vagrantup.com. On peut en chercher par OS :
- `generic/alpine317` : Alpine Linux 3.17 (~50 Mo — très légère !)
- `ubuntu/jammy64` : Ubuntu 22.04 LTS (~500 Mo)
- `debian/bullseye64` : Debian 11 (~300 Mo)

> **Pourquoi Alpine Linux ?**
> Alpine est la distribution Linux **la plus légère** qui existe. Elle est conçue pour les environnements aux ressources limitées. Son gestionnaire de paquets s'appelle `apk` (au lieu de `apt` pour Ubuntu/Debian). Elle utilise `musl libc` et `BusyBox` à la place de la `glibc` et des `GNU coreutils` habituels. Sa taille : ~5 Mo en conteneur Docker, ~30 Mo en VM.

## 2.2 Installer Vagrant et VirtualBox

### Sur Ubuntu / Debian

**VirtualBox :**
```bash
# Mettre à jour la liste des paquets
sudo apt update

# Installer VirtualBox et son module noyau
sudo apt install virtualbox virtualbox-dkms -y

# Vérifier l'installation
vboxmanage --version
# Résultat attendu : quelque chose comme "7.0.12r159484"
```

> **Qu'est-ce que `sudo` ?**
> `sudo` signifie "Super User DO" — ça permet d'exécuter une commande avec les droits administrateur (root). Sur Linux, seul root peut installer des logiciels système.

> **Qu'est-ce que `apt` ?**
> `apt` est le gestionnaire de paquets d'Ubuntu/Debian. Il télécharge et installe des logiciels depuis des dépôts officiels. C'est l'équivalent de l'App Store, mais en ligne de commande.

**Vagrant :**
```bash
# Télécharger et ajouter la clé GPG de HashiCorp (l'éditeur de Vagrant)
# GPG = cryptographie pour vérifier que le logiciel vient bien de HashiCorp
wget -O- https://apt.releases.hashicorp.com/gpg | \
  sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

# Ajouter le dépôt HashiCorp à la liste des sources APT
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list

# Mettre à jour et installer
sudo apt update && sudo apt install vagrant -y

# Vérifier
vagrant --version
# Résultat attendu : "Vagrant 2.x.x"
```

> **Qu'est-ce que GPG ?**
> GPG (GNU Privacy Guard) est un système de cryptographie. Quand on ajoute une clé GPG d'un éditeur, on dit à notre système "je fais confiance aux logiciels signés par cet éditeur". C'est une mesure de sécurité pour éviter d'installer des logiciels malveillants.

### Sur macOS

```bash
# Avec Homebrew (gestionnaire de paquets macOS)
brew install --cask virtualbox
brew install vagrant

# Vérifications
vboxmanage --version
vagrant --version
```

> **Si Homebrew n'est pas installé :**
> ```bash
> /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
> ```

### Sur Windows

1. Télécharger VirtualBox : https://www.virtualbox.org/wiki/Downloads → "Windows hosts"
2. Lancer l'installateur `.exe`, tout accepter par défaut
3. Télécharger Vagrant : https://developer.hashicorp.com/vagrant/downloads → Windows AMD64
4. Lancer l'installateur `.msi`, tout accepter
5. **Redémarrer Windows** (important !)
6. Ouvrir PowerShell et vérifier :
   ```powershell
   vagrant --version
   vboxmanage --version
   ```

> **Problème courant Windows : Hyper-V et VirtualBox**
> Windows peut avoir Hyper-V activé (pour WSL2, Docker Desktop...). VirtualBox et Hyper-V sont incompatibles. Si VirtualBox refuse de démarrer des VMs :
> - Ouvre PowerShell en admin
> - `bcdedit /set hypervisorlaunchtype off`
> - Redémarre Windows
> - Ou : active le mode "hypervisor platform" dans les fonctionnalités Windows (parfois compatible)

## 2.3 Anatomie complète d'un Vagrantfile

Le Vagrantfile est écrit en **Ruby**. Tu n'as pas besoin d'apprendre Ruby, mais quelques notions t'aideront à comprendre la syntaxe.

### Les bases de la syntaxe Ruby utilisée

```ruby
# Commentaire : commence par #

# Variable : pas de type explicite
NOM = "valeur"
NOMBRE = 42

# Interpolation de chaîne (mettre une variable dans du texte)
PRENOM = "Alice"
message = "Bonjour #{PRENOM} !"   # → "Bonjour Alice !"

# Bloc do...end (équivalent à { } dans d'autres langages)
quelquechose.faire("paramètre") do |variable|
  # code qui utilise 'variable'
end  # fin du bloc

# Chaîne multiligne
texte = <<-FIN
  Ligne 1
  Ligne 2
  Ligne 3
FIN
```

### Vagrantfile commenté ligne par ligne

```ruby
# -*- mode: ruby -*-
# vi: set ft=ruby :
# ↑ Ces deux lignes sont des commentaires spéciaux pour les éditeurs de texte
#   Ils demandent à vim/emacs d'activer la coloration syntaxique Ruby
#   Ce n'est pas obligatoire, mais c'est une bonne pratique

# ════════════════════════════════════════════════════════════════
# VARIABLES DE CONFIGURATION
# ════════════════════════════════════════════════════════════════

# IMPORTANT : Remplace "monlogin" par ton vrai login 42
LOGIN = "monlogin"

# On construit les noms à partir du login
# #{LOGIN} insère la valeur de LOGIN dans la chaîne
SERVER_NAME = "#{LOGIN}S"    # Ex: si LOGIN="john" → "johnS"
WORKER_NAME = "#{LOGIN}SW"   # Ex: "johnSW"

# IPs imposées par le sujet (NE PAS CHANGER)
SERVER_IP = "192.168.56.110"
WORKER_IP = "192.168.56.111"

# ════════════════════════════════════════════════════════════════
# DÉBUT DE LA CONFIGURATION
#
# Vagrant.configure("2") : utilise la version 2 de l'API Vagrant
#   "2" est un numéro de version de l'API, pas la version du logiciel
#   Toujours mettre "2" avec les versions modernes de Vagrant
#
# do |config| : ouvre un bloc de code
#   |config| = la variable qui représente la configuration globale
#   Tout ce qu'on fait avec 'config' s'applique à TOUTES les VMs
# ════════════════════════════════════════════════════════════════

Vagrant.configure("2") do |config|

  # ──────────────────────────────────────────────────────────────
  # config.vm.box
  # Définit l'image de base (box) pour toutes les VMs
  # "generic/alpine317" = Alpine Linux 3.17 hébergé sur vagrantcloud.com
  # La box est téléchargée automatiquement la première fois
  # ──────────────────────────────────────────────────────────────
  config.vm.box = "generic/alpine317"

  # ──────────────────────────────────────────────────────────────
  # config.vm.box_check_update
  # false = ne pas vérifier si une mise à jour de la box est disponible
  # À chaque vagrant up, Vagrant contacte vagrantcloud.com pour vérifier
  # En mettant false, on économise du temps (surtout en environnement sans internet)
  # ──────────────────────────────────────────────────────────────
  config.vm.box_check_update = false

  # ════════════════════════════════════════════════════════════════
  # DÉFINITION DE LA MACHINE 1 : SERVEUR K3s
  #
  # config.vm.define "nom" do |variable|
  #   Crée une VM nommée "nom"
  #   Tout ce qu'on fait avec 'variable' s'applique uniquement à cette VM
  # ════════════════════════════════════════════════════════════════

  config.vm.define SERVER_NAME do |server|
    # server = variable qui représente cette VM spécifique

    # ──────────────────────────────────────────────────────────
    # server.vm.hostname
    # Le nom d'hôte de la machine dans son propre OS Linux
    # Si tu tapes "hostname" dans la VM, tu vois cette valeur
    # Aussi utilisé pour les résolutions DNS locales
    # ──────────────────────────────────────────────────────────
    server.vm.hostname = SERVER_NAME

    # ──────────────────────────────────────────────────────────
    # server.vm.network
    # Configure l'interface réseau
    #
    # "private_network" = réseau privé hôte-seulement
    #   → La VM peut parler à l'hôte
    #   → Les VMs peuvent se parler entre elles
    #   → Internet ne peut PAS accéder directement à la VM
    #   → La VM PEUT accéder à internet (via NAT automatique sur eth0)
    #
    # ip: "192.168.56.110" = adresse IP fixe pour cette VM
    #   → Cette IP ne changera jamais, contrairement à une IP DHCP
    #   → 192.168.56.x est le réseau par défaut de VirtualBox "host-only"
    # ──────────────────────────────────────────────────────────
    server.vm.network "private_network", ip: SERVER_IP

    # ──────────────────────────────────────────────────────────
    # server.vm.provider
    # Configuration spécifique au provider VirtualBox
    # do |vb| : 'vb' représente la config VirtualBox
    # ──────────────────────────────────────────────────────────
    server.vm.provider "virtualbox" do |vb|

      # Nom affiché dans l'interface graphique de VirtualBox
      vb.name = SERVER_NAME

      # Mémoire RAM allouée en mégaoctets
      # 1024 = 1 Go de RAM
      # K3s en mode serveur a besoin d'au moins 512 Mo
      # 1024 est confortable pour éviter les problèmes
      vb.memory = 1024

      # Nombre de cœurs CPU virtuels alloués
      # 1 est suffisant pour ce projet
      vb.cpus = 1

    end  # fin provider virtualbox

    # ──────────────────────────────────────────────────────────
    # server.vm.provision
    # Script exécuté sur la VM après son démarrage initial
    #
    # "shell" = type de provision (script shell)
    # path: "scripts/server.sh" = chemin vers le script
    #   Ce chemin est RELATIF au Vagrantfile
    #   Dans la VM, le fichier se trouve à /vagrant/scripts/server.sh
    #   (car /vagrant est le dossier partagé avec le Vagrantfile)
    #
    # Le script est exécuté en tant que ROOT automatiquement
    # Il est exécuté UNE SEULE FOIS lors de la création initiale
    # Pour le relancer : vagrant provision ou vagrant reload --provision
    # ──────────────────────────────────────────────────────────
    server.vm.provision "shell", path: "scripts/server.sh"

  end  # fin define SERVER_NAME


  # ════════════════════════════════════════════════════════════════
  # DÉFINITION DE LA MACHINE 2 : WORKER K3s
  # ════════════════════════════════════════════════════════════════

  config.vm.define WORKER_NAME do |worker|

    worker.vm.hostname = WORKER_NAME

    # IP différente du serveur !
    worker.vm.network "private_network", ip: WORKER_IP

    worker.vm.provider "virtualbox" do |vb|
      vb.name   = WORKER_NAME
      # Le worker peut fonctionner avec moins de RAM
      # Car il n'héberge pas le control plane K3s
      vb.memory = 512
      vb.cpus   = 1
    end

    worker.vm.provision "shell", path: "scripts/worker.sh"

  end  # fin define WORKER_NAME

end  # fin Vagrant.configure
# ↑ Ce "end" ferme le bloc ouvert par "Vagrant.configure("2") do |config|"
```

## 2.4 Toutes les commandes Vagrant expliquées

### `vagrant up` — La commande principale

```bash
vagrant up
```

**Ce qui se passe en détail :**
1. Vagrant lit le Vagrantfile
2. Pour chaque VM définie :

   d. Attend que SSH soit disponible

   f. Configure le réseau

```bash
vagrant up                    # Démarre toutes les VMs
```

```bash
vagrant ssh loginS
```

**Pour sortir de la VM :**
```bash
exit    # ou Ctrl+D
```

## 2.5 Le dossier partagé /vagrant

## 2.6 Les réseaux Vagrant en détail

Vagrant configure deux interfaces réseau pour chaque VM :

### Interface 1 : NAT (Network Address Translation) — Automatique

```
VM ──[NAT]──> Machine hôte ──> Internet
```

Cette interface est créée **automatiquement** par Vagrant. Elle donne à la VM un accès à internet via ta connexion. Sans elle, la VM ne pourrait pas télécharger K3s ou des packages.

Elle a toujours l'IP `10.0.2.15` dans VirtualBox. C'est l'IP interne de NAT — pas utile pour notre projet.

### Interface 2 : Private Network — Ce qu'on configure

```
VM1 (192.168.56.110) ─┐
                       ├──> Réseau "host-only" virtuel ──> Ta machine
VM2 (192.168.56.111) ─┘
```

Cette interface est créée avec notre configuration :
```ruby
server.vm.network "private_network", ip: "192.168.56.110"
```

Caractéristiques :
- Les VMs peuvent se parler entre elles
- Ta machine hôte peut se connecter aux VMs (pour faire des curl tests)
- Internet ne peut PAS accéder directement aux VMs
- L'IP est fixe (ne change pas aux redémarrages)

### Tester la connectivité réseau

```bash
# Depuis ta machine hôte : tester que le serveur répond
ping 192.168.56.110
curl http://192.168.56.110    # Après installation de K3s et déploiement

# Depuis la VM serveur : tester que le worker est joignable
vagrant ssh loginS
ping 192.168.56.111

# Depuis la VM worker : tester que le serveur est joignable
vagrant ssh loginSW
ping 192.168.56.110
```

# CHAPITRE 5 — K3S : KUBERNETES ALLÉGÉ EN PROFONDEUR

## 5.1 Architecture de K3s

### Comment K3s est organisé

K3s est distribué comme un **unique binaire** de ~50 Mo qui contient tout :

```
┌─────────────────────────────────────────────────────────────────┐
│                    BINAIRE K3S (~50 Mo)                          │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │               COMPOSANTS CONTROL PLANE                   │   │
│  │  [API Server]  [Scheduler]  [Controller Manager]         │   │
│  │  [SQLite (remplace etcd)]                                │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │               COMPOSANTS WORKER                          │   │
│  │  [kubelet]  [kube-proxy]  [containerd]                   │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │               COMPOSANTS RÉSEAU                          │   │
│  │  [Flannel CNI]  [CoreDNS]  [Traefik Ingress]             │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │               STOCKAGE                                   │   │
│  │  [Local Path Provisioner]  [metrics-server]              │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### Modes d'installation

**Mode serveur (control plane + worker) :**
```
k3s server
├── Démarre le control plane (API Server, Scheduler, etc.)
├── Démarre les composants worker (kubelet, proxy)
├── Peut aussi exécuter des pods de travail
└── Génère le token de jonction pour les agents
```

**Mode agent (worker uniquement) :**
```
k3s agent
├── Démarre seulement les composants worker
├── Se connecte au serveur K3s via le token
└── Exécute les pods assignés par le scheduler
```

### Le token de jonction

```
SERVEUR                              AGENT
  │                                    │
  │ Génère /var/lib/rancher/            │
  │ k3s/server/node-token               │
  │                                    │
  │ ← On copie ce token pour l'agent   │
  │                                    │
  │                    K3S_URL + TOKEN  │
  │ ←────────────────────────────────  │
  │                                    │
  │ Vérifie le token                   │
  │ Accepte ou refuse l'agent           │
  │ ────────────────────────────────→  │
  │              Certificat TLS signé  │
  │              Agent enregistré !    │
```

Le token sert à **authentifier** les agents qui veulent rejoindre le cluster. Sans le bon token, un agent ne peut pas rejoindre.

### Flannel : le réseau des pods

Flannel est le **plugin réseau CNI (Container Network Interface)** utilisé par K3s. Il permet aux pods sur des nœuds différents de communiquer.

```
NŒUD 1 (192.168.56.110)          NŒUD 2 (192.168.56.111)
  Pod A (IP: 10.42.0.5)            Pod B (IP: 10.42.1.7)
      │                                  │
  Flannel sur nœud 1               Flannel sur nœud 2
      │                                  │
  ────────────────────────────────────────
  Réseau physique (192.168.56.x)
  
Pod A veut parler à Pod B :
  Pod A → 10.42.1.7
  Flannel encapsule dans un tunnel VXLAN vers 192.168.56.111
  Flannel sur nœud 2 décapsule
  Livre au Pod B → Pod B reçoit le paquet de Pod A
```

### CoreDNS : la résolution DNS interne

CoreDNS permet aux pods de se parler par **nom** au lieu d'IP.

```
Pod A veut accéder au Service "mon-service" dans le namespace "default" :
  → DNS query: "mon-service.default.svc.cluster.local"
  → CoreDNS répond: "10.96.45.123" (ClusterIP du Service)
  → Pod A connecte à 10.96.45.123

Format DNS complet dans K8s :
[service-name].[namespace].svc.[cluster-domain]
mon-service.default.svc.cluster.local

Raccourcis qui fonctionnent aussi (dans le même namespace) :
mon-service
mon-service.default
```

### Traefik : l'Ingress Controller intégré

Traefik est installé automatiquement par K3s. Il tourne dans le namespace `kube-system`.

```bash
# Vérifier que Traefik tourne
kubectl get pods -n kube-system | grep traefik
# traefik-xxxx   1/1   Running   0   5m

# Voir la configuration de Traefik
kubectl get deployment -n kube-system traefik -o yaml
```

**Comment Traefik lit les Ingress :**
```
1. Traefik se connecte à l'API Kubernetes au démarrage
2. Il "watch" (surveille) les objets Ingress via l'API
3. Quand tu crées un Ingress → Traefik le détecte immédiatement
4. Traefik met à jour sa table de routage interne
5. Les nouvelles requêtes sont routées selon les nouvelles règles
```

## 5.2 Installation de K3s en mode serveur

### La commande d'installation expliquée mot par mot

```bash
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="--write-kubeconfig-mode=644 --node-ip=192.168.56.110 --bind-address=192.168.56.110 --advertise-address=192.168.56.110" sh -
```

Décomposons cette commande :

```
curl
  → outil en ligne de commande pour faire des requêtes HTTP

  -s   (silent)
  → pas d'affichage de progression

  -f   (fail)
  → retourne une erreur si le serveur répond avec un code d'erreur HTTP

  -L   (location)
  → suit les redirections HTTP (si le serveur dit "va là-bas")

https://get.k3s.io
  → URL du script d'installation officiel K3s

|  (pipe)
  → envoie la sortie de curl vers la commande suivante

INSTALL_K3S_EXEC="..."
  → variable d'environnement qui passe des arguments à K3s lors de l'installation
  → ces arguments deviennent les flags de lancement de k3s server

sh -
  → exécute le script reçu via le pipe
  → le "-" signifie "lire le script depuis l'entrée standard (stdin)"

ARGUMENTS K3S :
--write-kubeconfig-mode=644
  → le fichier kubeconfig (/etc/rancher/k3s/k3s.yaml) sera lisible par tous
  → sans ça, seul root peut utiliser kubectl
  → 644 = rw-r--r-- (propriétaire peut lire/écrire, autres peuvent lire)

--node-ip=192.168.56.110
  → l'IP que ce nœud annonce aux autres membres du cluster
  → TRÈS important : sans ça, K3s utiliserait l'IP NAT (10.0.2.15)
  → et les agents ne pourraient pas se connecter depuis l'extérieur de la VM

--bind-address=192.168.56.110
  → l'adresse IP sur laquelle l'API Server écoute
  → avec ça, l'API Server écoute sur 192.168.56.110:6443
  → sans ça, il écouterait sur 0.0.0.0 (toutes les interfaces) — moins sécurisé

--advertise-address=192.168.56.110
  → l'adresse annoncée dans les certificats TLS
  → les agents vérifient que l'adresse du serveur correspond aux certificats
  → doit être la même IP que celle utilisée dans K3S_URL côté agent
```

### Ce qui se passe pendant l'installation

```bash
# L'installation fait ces étapes automatiquement :

# 1. Télécharge le binaire k3s depuis GitHub releases
#    (~50 Mo, peut prendre 1-2 min selon la connexion)

# 2. Place le binaire dans /usr/local/bin/k3s

# 3. Crée des liens symboliques :
#    /usr/local/bin/kubectl → k3s kubectl
#    /usr/local/bin/crictl  → k3s crictl
#    /usr/local/bin/ctr     → k3s ctr

# 4. Crée le service systemd /etc/systemd/system/k3s.service
# Ce service démarre K3s automatiquement au boot de la VM

# 5. Démarre le service k3s

# 6. Génère les certificats TLS

# 7. Crée le fichier kubeconfig /etc/rancher/k3s/k3s.yaml

# 8. Génère le token de jonction /var/lib/rancher/k3s/server/node-token

# 9. Démarre Flannel, CoreDNS, Traefik, metrics-server automatiquement
```

### Vérifier l'état après installation

```bash
# Sur la VM serveur :

# 1. K3s service est-il actif ?
sudo systemctl status k3s
# Cherche : "Active: active (running)"

# 2. Les nœuds sont-ils prêts ?
kubectl get nodes
# loginS   Ready   control-plane,master   5m   v1.27.x+k3s1
# (le worker n'apparaît que quand il aura rejoint)

# 3. Les pods système tournent-ils ?
kubectl get pods -n kube-system
# coredns-xxxx           1/1   Running
# local-path-provisioner  1/1   Running
# metrics-server-xxxx    1/1   Running
# traefik-xxxx           1/1   Running

# 4. Le token est-il généré ?
cat /var/lib/rancher/k3s/server/node-token
# Doit afficher une longue chaîne : K10abc...::server:xyz...

# 5. Le kubeconfig est-il accessible ?
cat /etc/rancher/k3s/k3s.yaml
# Doit afficher la config kubectl avec l'IP 192.168.56.110
```

## 5.3 Installation de K3s en mode agent

### La commande d'installation de l'agent

```bash
curl -sfL https://get.k3s.io | \
    K3S_URL="https://192.168.56.110:6443" \
    K3S_TOKEN="$(cat /vagrant/node-token)" \
    INSTALL_K3S_EXEC="--node-ip=192.168.56.111" \
    sh -
```

**Explication des variables :**
```
K3S_URL="https://192.168.56.110:6443"
  → Adresse de l'API Server K3s du serveur
  → L'agent va se connecter à cette adresse pour recevoir ses instructions
  → Port 6443 = port standard de l'API Kubernetes

K3S_TOKEN="..."
  → Le token de sécurité récupéré sur le serveur
  → Sans le bon token, l'agent est refusé
  → Equivalent d'un mot de passe pour rejoindre le cluster

INSTALL_K3S_EXEC="--node-ip=192.168.56.111"
  → IP de CE nœud worker (pas du serveur !)
  → L'agent s'annonce au serveur avec cette IP
  → Les pods placés sur ce nœud sont accessibles via 192.168.56.111
```

### Ce que fait l'agent en s'enregistrant

```
1. Agent se connecte à https://192.168.56.110:6443
2. Présente le token pour s'authentifier
3. Serveur vérifie le token → OK
4. Serveur génère un certificat TLS signé pour cet agent
5. Agent reçoit sa configuration (ClusterDNS, ClusterDomain...)
6. Agent démarre kubelet et kube-proxy
7. kubelet s'enregistre auprès de l'API Server
8. Le nœud apparaît dans kubectl get nodes avec le statut "Ready"

Durée typique : 30-60 secondes
```

```ruby
  config.vm.box_check_update = false

  # ─── VM 1 : SERVEUR K3S (control plane) ─────────────────────────

      vb.customize ["modifyvm", :id, "--natdnshostresolver1", "on"]  # DNS via hôte
      vb.customize ["modifyvm", :id, "--ioapic", "on"]               # I/O APIC activé
    end

    # Script qui installe K3s en mode serveur
    server.vm.provision "shell", path: "scripts/server.sh"

  end

  # ─── VM 2 : WORKER K3S (agent) ──────────────────────────────────
  config.vm.define WORKER_NAME do |worker|

    worker.vm.hostname = WORKER_NAME
    worker.vm.network "private_network", ip: WORKER_IP

    worker.vm.provider "virtualbox" do |vb|
      vb.name   = WORKER_NAME
      vb.memory = 512    # 512 Mo suffisent pour un agent K3s
      vb.cpus   = 1
      vb.customize ["modifyvm", :id, "--natdnshostresolver1", "on"]
      vb.customize ["modifyvm", :id, "--ioapic", "on"]
    end

    # Script qui installe K3s en mode agent
    worker.vm.provision "shell", path: "scripts/worker.sh"

  end

end
```

```bash
# │                                                                 │

# │ Ne PAS utiliser #!/bin/bash : bash n'est pas installé par        │
# │ défaut sur Alpine Linux !                                        │

# Si une commande échoue (code de retour ≠ 0), le script s'arrête immédiatement

# "apk update" met à jour la liste des paquets disponibles
# C'est comme "apt update" sur Ubuntu/Debian
echo ">>> Mise à jour des paquets Alpine..."

# ─── ÉTAPE 2 : INSTALLATION DE CURL ───────────────────────────────
# curl est l'outil qui télécharge le script d'installation K3s

# - Initialiser la base de données SQLite
```

## 6.4 Script du worker : ligne par ligne

```bash
nano ~/IoT-project/p1/scripts/worker.sh
```

```bash
#!/bin/sh
# ┌─────────────────────────────────────────────────────────────────┐
# │ SCRIPT D'INSTALLATION DU WORKER K3S                             │
# │                                                                 │
# │ Ce script est exécuté sur la VM worker après son démarrage.     │
# │ Il attend que le serveur ait créé le token, puis installe        │
# │ K3s en mode agent (worker).                                      │
# └─────────────────────────────────────────────────────────────────┘

set -e

SERVER_IP="192.168.56.110"
WORKER_IP="192.168.56.111"

echo "================================================================"
echo "  Installation K3s — Mode WORKER (Agent)"
echo "  IP: ${WORKER_IP}"
echo "  Serveur: ${SERVER_IP}"
echo "================================================================"

# ─── ÉTAPE 1 : PRÉREQUIS ──────────────────────────────────────────
echo ">>> Mise à jour des paquets et installation de curl..."
apk update
apk add curl --no-cache

# ─── ÉTAPE 2 : ATTENDRE LE TOKEN DU SERVEUR ───────────────────────
# Le serveur crée le token lors de son installation.
# Le worker doit attendre que le token soit disponible.
# Si le worker démarre AVANT le serveur, il ne trouverait pas le token.
#
# /vagrant/ est le dossier partagé entre toutes les VMs.
# Le serveur y copie le token, le worker le lit.

echo ">>> Attente du token de jonction..."
echo "    (Le serveur doit finir son installation d'abord)"

TIMEOUT=300    # Attendre au maximum 5 minutes
ELAPSED=0
TOKEN_PATH="/vagrant/node-token"

while [ ! -f "$TOKEN_PATH" ]; do
    # Le fichier n'existe pas encore → attendre
    echo "    Token pas encore disponible... (${ELAPSED}s / ${TIMEOUT}s)"
    sleep 5
    ELAPSED=$((ELAPSED + 5))

    # Si on a attendu trop longtemps → erreur
    if [ $ELAPSED -ge $TIMEOUT ]; then
        echo "ERREUR TIMEOUT: Le token n'est jamais apparu après ${TIMEOUT}s"
        echo "Vérifiez que le serveur a bien démarré :"
        echo "  vagrant ssh ${SERVER_IP%.*}S  # (remplace par ton login)"
        echo "  sudo systemctl status k3s"
        exit 1
    fi
done

echo ">>> Token trouvé ! (après ${ELAPSED} secondes)"

# Lire le contenu du token (enlève le saut de ligne final avec tr)
NODE_TOKEN=$(cat "$TOKEN_PATH" | tr -d '\n')

# Vérifier que le token n'est pas vide
if [ -z "$NODE_TOKEN" ]; then
    echo "ERREUR: Le fichier token est vide !"
    exit 1
fi

echo ">>> Token valide (longueur: $(echo -n "$NODE_TOKEN" | wc -c) caractères)"

# ─── ÉTAPE 3 : TESTER LA CONNECTIVITÉ VERS LE SERVEUR ─────────────
echo ">>> Test de connectivité vers le serveur..."
if ping -c 2 "$SERVER_IP" > /dev/null 2>&1; then
    echo ">>> Serveur joignable !"
else
    echo "ERREUR: Impossible de pinger le serveur ${SERVER_IP}"
    echo "Vérifiez la configuration réseau."
    exit 1
fi

# Tester aussi que l'API K3s répond
echo ">>> Vérification que l'API K3s du serveur répond..."
RETRIES=0
until curl -sk "https://${SERVER_IP}:6443/healthz" | grep -q "ok"; do
    RETRIES=$((RETRIES + 1))
    if [ $RETRIES -ge 12 ]; then
        echo "ERREUR: L'API K3s du serveur ne répond pas !"
        exit 1
    fi
    echo "    API pas encore prête... (tentative ${RETRIES}/12)"
    sleep 5
done
echo ">>> API K3s du serveur opérationnelle !"

# ─── ÉTAPE 4 : INSTALLATION DE K3S EN MODE AGENT ──────────────────
echo ">>> Installation de K3s en mode AGENT..."
echo "    Connexion au serveur : https://${SERVER_IP}:6443"

curl -sfL https://get.k3s.io | \
    K3S_URL="https://${SERVER_IP}:6443" \
    K3S_TOKEN="${NODE_TOKEN}" \
    INSTALL_K3S_EXEC="--node-ip=${WORKER_IP}" \
    sh -

# ─── ÉTAPE 5 : VÉRIFIER L'ENREGISTREMENT ──────────────────────────
echo ">>> K3s agent installé !"
echo ">>> Attente de l'enregistrement dans le cluster (30 secondes)..."
sleep 30

echo ">>> État du service k3s-agent :"
sudo systemctl status k3s-agent --no-pager | head -15
# Doit afficher : "Active: active (running)"

# ─── RÉSUMÉ FINAL ────────────────────────────────────────────────
echo ""
echo "================================================================"
echo "  Worker K3s installé avec succès !"
echo "================================================================"
echo "  Récapitulatif :"
```

# CHAPITRE 7 — L'INGRESS ET LE ROUTAGE HTTP EN PROFONDEUR

## 7.1 Le parcours complet d'une requête HTTP

Comprendre exactement ce qui se passe quand tu fais :
```bash
curl -H "Host:app1.com" http://192.168.56.110
```

### Étape 1 : La requête quitte ta machine

```
Ta machine :
  curl crée une connexion TCP vers 192.168.56.110:80
  Envoie la requête HTTP :
  
  GET / HTTP/1.1
  Host: app1.com
  User-Agent: curl/7.88.1
  Accept: */*
  Connection: Keep-Alive
```

### Étape 2 : Traefik reçoit la requête

```
Traefik écoute sur 192.168.56.110:80
  ↓
Traefik lit les en-têtes HTTP :
  Host: app1.com   ← C'est cet en-tête qui détermine le routing !
  ↓
Traefik consulte sa table de routage (construite depuis les objets Ingress) :
  
  Table de routage Traefik :
  ┌──────────────────────────────────────────────────────────────┐
  │ Host: app1.com → Service "app-one" (port 80)                 │
  │ Host: app2.com → Service "app-two" (port 80)                 │
  │ (défaut)       → Service "app-three" (port 80)               │
  └──────────────────────────────────────────────────────────────┘
  ↓
  Trouve la règle : Host: app1.com → Service "app-one"
```

### Étape 3 : Le Service résout les endpoints

```
Service "app-one" (ClusterIP: 10.96.x.x, port 80) :
  ↓
kube-proxy a maintenu la liste des pods du Service :
  Endpoints de "app-one" :
  - 10.42.0.15:80 (pod app-one-xxxxx)
  
  (si 3 réplicas comme app-two :)
  - 10.42.0.20:80 (pod app-two-xxxxx-1)
  - 10.42.0.21:80 (pod app-two-xxxxx-2)
  - 10.42.0.22:80 (pod app-two-xxxxx-3)
  ↓
Sélection round-robin : 10.42.0.15:80
```

### Étape 4 : Le pod répond

```
Pod app-one (10.42.0.15:80) :
  nginx reçoit la requête
  Lit /usr/share/nginx/html/index.html
  Renvoie HTTP 200 + le contenu HTML
  ↓
Service → Traefik → Ta machine
```

### Étape 5 : Tu vois la réponse

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
...

<!DOCTYPE html>
<html>
  <h1>Hello from app1.</h1>
  ...
</html>
```

## 7.2 Les règles Ingress : cas avancés

### Routage par chemin URL

```yaml
# En plus du routing par Host, on peut router par PATH
spec:
  rules:
  - host: monsite.com
    http:
      paths:
      # /api/* → service API
      - path: /api
        pathType: Prefix      # /api, /api/users, /api/products → API
        backend:
          service:
            name: api-service
            port:
              number: 3000
      
      # /static/* → service de fichiers statiques
      - path: /static
        pathType: Prefix
        backend:
          service:
            name: static-service
            port:
              number: 80
      
      # Tout le reste → application web principale
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
```

### Les types de pathType

```
Prefix :
  Le chemin correspond si la requête COMMENCE par ce préfixe
  path: /api
  → /api → ✅
  → /api/users → ✅
  → /api/products/123 → ✅
  → /apitest → ❌ (doit être /api/ ou exactement /api)

Exact :
  Le chemin doit correspondre EXACTEMENT
  path: /api
  → /api → ✅
  → /api/ → ❌
  → /api/users → ❌

ImplementationSpecific :
  L'Ingress Controller décide (comportement Traefik-spécifique)
```

### `confs/ingress.yaml` — Le routeur HTTP

```yaml
# ═══════════════════════════════════════════════════════════════════
# INGRESS — Configuration du routage HTTP
#
# Cet objet est lu par Traefik (l'Ingress Controller de K3s).
# Il définit les règles de routage basées sur l'en-tête HTTP "Host".
# ═══════════════════════════════════════════════════════════════════

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: apps-ingress
  namespace: default
  annotations:
    # Indique quel Ingress Controller utiliser
    # K3s utilise Traefik par défaut
    kubernetes.io/ingress.class: "traefik"

spec:
  rules:

  # ─── RÈGLE 1 : Host app1.com ────────────────────────────────────
  # Si la requête HTTP a l'en-tête "Host: app1.com"
  # → Envoyer vers le Service "app-one"
  - host: app1.com
    http:
      paths:
      - path: /
        pathType: Prefix         # Correspond à "/" et tout chemin qui commence par "/"
        backend:
          service:
            name: app-one        # ← Doit correspondre exactement au nom du Service !
            port:
              number: 80

  # ─── RÈGLE 2 : Host app2.com ────────────────────────────────────
  - host: app2.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app-two
            port:
              number: 80

  # ─── RÈGLE PAR DÉFAUT : Aucun Host ──────────────────────────────
  # Cette règle ne spécifie PAS de "host"
  # Elle correspond à TOUTES les requêtes qui ne correspondent à aucune règle host
  # DOIT ÊTRE EN DERNIER (ordre d'évaluation de haut en bas)
  - http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app-three
            port:
              number: 80
```

## 8.5 Script de déploiement

```bash
#!/bin/sh
set -e

echo "================================================================"
echo "  Déploiement des 3 applications — Partie 2"
echo "================================================================"

# ─── ATTENTE QUE K3S SOIT PRÊT ────────────────────────────────────
echo ">>> Vérification que K3s est prêt..."
until kubectl get nodes 2>/dev/null | grep -q "Ready"; do
    echo "    Attente de K3s..."
    sleep 3
done

# ─── ATTENTE QUE TRAEFIK SOIT PRÊT ────────────────────────────────
echo ">>> Attente que Traefik soit opérationnel..."
WAIT=0
until kubectl get pods -n kube-system 2>/dev/null | grep "^traefik" | grep -q "Running"; do
    echo "    Traefik pas encore prêt... ($WAIT secondes)"
    sleep 5
    WAIT=$((WAIT + 5))
    if [ $WAIT -ge 120 ]; then
        echo "ERREUR: Traefik n'a pas démarré après 2 minutes"
        kubectl get pods -n kube-system
        exit 1
    fi
done
echo ">>> Traefik prêt !"

# ─── DÉPLOIEMENT DES APPLICATIONS ─────────────────────────────────
# kubectl apply -f applique le fichier YAML :
# - Si la ressource n'existe pas → la crée
# - Si la ressource existe → la met à jour (si changements)
# C'est l'approche "idempotente" : peut être appelée plusieurs fois sans problème

echo ">>> Déploiement App 1 (1 réplica)..."
kubectl apply -f /vagrant/confs/app1.yaml

echo ">>> Déploiement App 2 (3 réplicas)..."
kubectl apply -f /vagrant/confs/app2.yaml

echo ">>> Déploiement App 3 (1 réplica, défaut)..."
kubectl apply -f /vagrant/confs/app3.yaml

echo ">>> Configuration de l'Ingress..."
kubectl apply -f /vagrant/confs/ingress.yaml

# ─── ATTENTE QUE LES PODS SOIENT PRÊTS ────────────────────────────
echo ">>> Attente que tous les pods démarrent..."
echo "    (Kubernetes doit télécharger l'image nginx:alpine si pas en cache)"
echo "    (Cela peut prendre 1-3 minutes la première fois)"

# Attendre que les déploiements soient disponibles
kubectl rollout status deployment/app-one   --timeout=180s
kubectl rollout status deployment/app-two   --timeout=180s
kubectl rollout status deployment/app-three --timeout=180s

echo ">>> Tous les pods sont prêts !"

# ─── AFFICHAGE DE L'ÉTAT ──────────────────────────────────────────
echo ""
echo "=== ÉTAT DES PODS ==="
kubectl get pods -o wide
# Doit afficher : 1 pod app-one, 3 pods app-two, 1 pod app-three = 5 pods total

echo ""
echo "=== ÉTAT DES SERVICES ==="
kubectl get services
# Doit afficher : kubernetes (system), app-one, app-two, app-three

echo ""
echo "=== INGRESS (IMPORTANT POUR LA DÉFENSE) ==="
kubectl get ingress
# Doit afficher : apps-ingress avec les hosts

echo ""
echo "=== TESTS DE CONNECTIVITÉ ==="

# Test App 1
echo -n ">>> App 1 (Host: app1.com) : "
RESULT=$(curl -s -H "Host:app1.com" http://192.168.56.110 | grep -o "Hello from app." || echo "ERREUR")
echo "$RESULT"

# Test App 2
echo -n ">>> App 2 (Host: app2.com) : "
RESULT=$(curl -s -H "Host:app2.com" http://192.168.56.110 | grep -o "Hello from app." || echo "ERREUR")
echo "$RESULT"

# Test App 3 (défaut)
echo -n ">>> App 3 (défaut) : "
RESULT=$(curl -s http://192.168.56.110 | grep -o "Hello from app." || echo "ERREUR")
echo "$RESULT"

echo ""
echo "================================================================"
echo "  Déploiement terminé !"
echo "================================================================"
echo ""
echo "  Pour tester depuis ta machine hôte :"
echo "  curl -H 'Host:app1.com' http://192.168.56.110"
echo "  curl -H 'Host:app2.com' http://192.168.56.110"
echo "  curl http://192.168.56.110"
echo ""
echo "  IMPORTANT pour la défense : montre l'Ingress avec :"
echo "  kubectl get ingress"
echo "  kubectl describe ingress apps-ingress"
echo "================================================================"
```

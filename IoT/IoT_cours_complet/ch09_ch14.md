# CHAPITRE 9 — DOCKER EN PROFONDEUR

## 9.1 Qu'est-ce que Docker vraiment ?

Docker est une **plateforme complète** pour construire, distribuer et exécuter des conteneurs. Il est composé de plusieurs pièces :

```
┌──────────────────────────────────────────────────────────────────────────┐
│                         ÉCOSYSTÈME DOCKER                                 │
│                                                                          │
│  ┌──────────────────┐    ┌──────────────────┐    ┌──────────────────┐   │
│  │  docker CLI      │    │   Docker Hub     │    │   Dockerfile     │   │
│  │  (la commande    │    │   (registre      │    │   (recette pour  │   │
│  │   docker)        │    │   d'images)      │    │   construire     │   │
│  └────────┬─────────┘    └──────────────────┘    │   une image)     │   │
│           │ parle à                               └──────────────────┘   │
│  ┌────────▼─────────────────────────────────────────────────────────┐   │
│  │                    DOCKER DAEMON (dockerd)                        │   │
│  │                                                                  │   │
│  │   Reçoit les commandes, gère les images, conteneurs, réseaux,    │   │
│  │   volumes. C'est le "cerveau" de Docker.                         │   │
│  │                                                                  │   │
│  │   ┌──────────────────────────────────────────────────────────┐   │   │
│  │   │                   CONTAINERD                             │   │   │
│  │   │   Gère le cycle de vie des conteneurs                    │   │   │
│  │   │   ┌──────────────────────────────────────────────────┐   │   │   │
│  │   │   │   runc — Lance réellement les conteneurs Linux   │   │   │   │
│  │   │   └──────────────────────────────────────────────────┘   │   │   │
│  │   └──────────────────────────────────────────────────────────┘   │   │
│  └──────────────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────────┘
```

### L'analogie de la cuisine industrielle

Pense à Docker comme à une **cuisine industrielle professionnelle** :

- Le **Dockerfile** = la recette écrite
- L'**image Docker** = la recette préparée, mise en boîte (conserve)
- Le **conteneur** = la boîte ouverte, le plat en train d'être servi
- **Docker Hub** = le supermarché où on trouve des milliers de boîtes prêtes
- Le **Docker daemon** = le chef de cuisine qui suit les recettes

## 9.2 Installer Docker

### Ubuntu / Debian

```bash
# ── Étape 1 : Supprimer d'anciennes versions si elles existent ──
sudo apt remove docker docker-engine docker.io containerd runc 2>/dev/null || true

# ── Étape 2 : Installer les dépendances pour HTTPS ──────────────
sudo apt update
sudo apt install -y \
    ca-certificates \     # Certificats SSL
    curl \                # Pour télécharger
    gnupg \               # Pour vérifier les signatures
    lsb-release           # Pour obtenir le nom de la distrib Ubuntu

# ── Étape 3 : Ajouter la clé GPG officielle de Docker ───────────
# On dit à notre système : "je fais confiance à Docker Inc."
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
    sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# ── Étape 4 : Ajouter le dépôt Docker ───────────────────────────
# "$(dpkg --print-architecture)" = amd64 sur un PC standard
# "$(lsb_release -cs)" = jammy pour Ubuntu 22.04, focal pour 20.04, etc.
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# ── Étape 5 : Installer Docker ──────────────────────────────────
sudo apt update
sudo apt install -y \
    docker-ce \           # Docker Community Edition (le moteur)
    docker-ce-cli \       # La commande "docker"
    containerd.io \       # Le runtime de conteneurs
    docker-compose-plugin # Plugin Docker Compose

# ── Étape 6 : Permettre à ton utilisateur d'utiliser Docker ─────
# Sans ça, tu devrais faire "sudo docker ..." à chaque fois
sudo usermod -aG docker $USER
# ⚠️ Cette commande ne prend effet qu'après une NOUVELLE connexion
# Pour appliquer immédiatement dans le terminal courant :
newgrp docker

# ── Vérifications ────────────────────────────────────────────────
docker --version
# Docker version 24.x.x, build xxxxx

docker run hello-world
# Si tu vois "Hello from Docker!" → Docker fonctionne !
```

> **Que fait `sudo usermod -aG docker $USER` ?**
> - `usermod` = modifier un utilisateur
> - `-a` = ajouter (append) sans enlever les groupes existants
> - `-G docker` = ajouter au groupe "docker"
> - `$USER` = ton nom d'utilisateur courant
>
> Le groupe "docker" a accès au socket Docker (`/var/run/docker.sock`), ce qui permet de lancer des conteneurs sans `sudo`.

### macOS

```bash
# Méthode 1 : Docker Desktop (interface graphique + CLI)
brew install --cask docker
# Lance Docker Desktop depuis les Applications

# Méthode 2 : Juste le CLI (si tu veux du Docker léger)
```

## 9.3 Les images Docker — Tout comprendre

### Trouver des images sur Docker Hub

Docker Hub (https://hub.docker.com) est le registre public d'images. Il contient :
- Les **images officielles** : `nginx`, `ubuntu`, `alpine`, `node`, `python`...
- Les **images de la communauté** : `wil42/playground`, `argoproj/argocd`...

```bash
# Chercher une image
docker search nginx
# NAME                               DESCRIPTION                   STARS
# nginx                              Official build of Nginx.      19234
# bitnami/nginx                      Bitnami container image...    181
# ubuntu/nginx                       Ubuntu with Nginx             98

# Télécharger une image
docker pull nginx:alpine
# alpine : tag (version)
# sans tag = latest par défaut

docker pull nginx           # Télécharge nginx:latest
docker pull nginx:alpine    # Version Alpine (plus légère)
docker pull nginx:1.25.3    # Version spécifique

# Lister les images téléchargées
docker images
# REPOSITORY    TAG       IMAGE ID       CREATED       SIZE
# nginx         alpine    abc123def456   2 weeks ago   23.4MB
# ubuntu        22.04     def456abc789   3 weeks ago   77.8MB
```

### Anatomie d'un tag d'image

```
wil42/playground:v1
│     │           │
│     │           └── Tag (version, variante)
│     │               - "latest" = dernière version
│     │               - "alpine" = basé sur Alpine Linux
│     │               - "v1", "v2" = versions applicatives
│     │               - "1.25.3" = version exacte
│     │
│     └── Nom de l'image
│
└── Propriétaire (username Docker Hub)
    - Absent = image officielle Docker
    - Présent = image d'un utilisateur/organisation

Exemples :
nginx             = Image officielle nginx (latest)
nginx:alpine      = Image officielle nginx, variante Alpine
wil42/playground:v1  = Image de l'utilisateur wil42, tag v1
ghcr.io/user/img:tag = GitHub Container Registry
```

## 9.4 Construire ses propres images : le Dockerfile

### Structure d'un Dockerfile

```dockerfile
# ── FROM ──────────────────────────────────────────────────────────
# Toujours la PREMIÈRE instruction (sauf ARG avant FROM)
# Définit l'image de base
FROM alpine:3.18
# FROM node:18-alpine      ← Node.js sur Alpine
# FROM ubuntu:22.04        ← Ubuntu complet
# FROM scratch             ← Aucune base (image vide, pour les binaires statiques)

# ── ARG ───────────────────────────────────────────────────────────
# Variables de construction (disponibles SEULEMENT pendant le build)
# Différent de ENV (disponibles aussi pendant l'exécution)
ARG VERSION=1.0
ARG BUILD_DATE

# ── ENV ───────────────────────────────────────────────────────────
# Variables d'environnement disponibles dans le conteneur
ENV APP_PORT=8888
ENV APP_VERSION=${VERSION}
ENV NODE_ENV=production

# ── WORKDIR ───────────────────────────────────────────────────────
# Définit le répertoire de travail pour les instructions suivantes
# Équivalent à "cd /app" + "mkdir -p /app" si n'existe pas
WORKDIR /app

# ── COPY ──────────────────────────────────────────────────────────
# Copie des fichiers depuis le contexte de build vers l'image
COPY package.json .            # Copie package.json dans /app/
COPY package*.json ./          # Copie package.json et package-lock.json
COPY . .                       # Copie TOUT dans le répertoire courant
COPY src/ /app/src/            # Copie le dossier src/

# ── ADD ───────────────────────────────────────────────────────────
# Comme COPY mais avec des super-pouvoirs :
# - Peut extraire des archives .tar.gz automatiquement
# - Peut télécharger depuis des URLs
# En pratique : utiliser COPY par défaut, ADD seulement si besoin
ADD mon-archive.tar.gz /app/   # Extrait l'archive dans /app/
ADD https://example.com/file /app/file  # Télécharge

# ── RUN ───────────────────────────────────────────────────────────
# Exécute des commandes dans l'image pendant la construction
# Chaque RUN crée une nouvelle couche → grouper les commandes !
RUN apk update && apk add --no-cache curl nginx

# ❌ MAUVAIS : deux couches inutiles
RUN apk update
RUN apk add curl

# ✅ BON : une seule couche
RUN apk update && \
    apk add --no-cache \
    curl \
    nginx \
    wget

# ── EXPOSE ────────────────────────────────────────────────────────
# Documente le port que le conteneur utilise
# ⚠️ N'ouvre PAS réellement le port ! C'est de la documentation.
# C'est docker run -p 8080:80 qui ouvre vraiment le port
EXPOSE 8888

# ── USER ──────────────────────────────────────────────────────────
# Change l'utilisateur pour les instructions suivantes et pour l'exécution
# Bonne pratique de sécurité : ne pas tourner en root !
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# ── VOLUME ────────────────────────────────────────────────────────
# Déclare un point de montage pour les données persistantes
VOLUME ["/app/data"]

# ── CMD ───────────────────────────────────────────────────────────
# La commande par défaut quand le conteneur démarre
# Peut être remplacée par docker run IMAGE autre-commande
# Forme recommandée : tableau JSON (exec form)
CMD ["node", "app.js"]
CMD ["nginx", "-g", "daemon off;"]

# Forme shell (moins recommandée car lance un sous-shell)
CMD node app.js

# ── ENTRYPOINT ────────────────────────────────────────────────────
# Comme CMD mais plus difficile à remplacer
# Souvent utilisé pour les scripts d'initialisation
ENTRYPOINT ["/app/entrypoint.sh"]
# Avec ENTRYPOINT, CMD devient les arguments par défaut
```

### Créer l'image de l'application du projet (alternative à wil42/playground)

Si tu veux créer ta propre application au lieu d'utiliser `wil42/playground` :

```
mon-app/
├── Dockerfile
├── app.js
└── package.json
```

**`package.json`** :
```json
{
  "name": "iot-playground",
  "version": "1.0.0",
  "description": "Application IoT pour le projet 42",
  "main": "app.js",
  "scripts": {
    "start": "node app.js"
  },
  "dependencies": {}
}
```

**`app.js`** :
```javascript
// Application Node.js ultra-simple
// Répond avec un JSON contenant la version
const http = require('http');

// La version est définie via une variable d'environnement
// Cela permet de changer la version SANS modifier le code
const VERSION  = process.env.APP_VERSION || 'v1';
const PORT     = process.env.PORT        || 8888;
const HOSTNAME = process.env.HOSTNAME    || 'unknown';

const server = http.createServer((req, res) => {
  // Log de chaque requête (visible dans kubectl logs)
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);

  // Réponse JSON
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify({
    status:    'ok',
    message:   VERSION,
    pod:       HOSTNAME,
    timestamp: new Date().toISOString()
  }));
});

server.listen(PORT, () => {
  console.log(`Application ${VERSION} démarrée sur le port ${PORT}`);
});
```

**`Dockerfile`** :
```dockerfile
# Image de base légère
FROM node:18-alpine

# Bonne pratique : créer un utilisateur non-root
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Répertoire de travail
WORKDIR /app

# Copier et installer les dépendances
# On copie package.json en PREMIER pour profiter du cache Docker :
# Si le code change mais pas package.json → pas besoin de réinstaller
COPY package*.json ./
RUN npm install --production

# Copier le code source
COPY app.js .

# Changer le propriétaire des fichiers
RUN chown -R appuser:appgroup /app

# Utiliser l'utilisateur non-root
USER appuser

# Port d'écoute
EXPOSE 8888

# Version par défaut (peut être surchargée au build ou au runtime)
ENV APP_VERSION=v1

# Démarrer l'application
CMD ["node", "app.js"]
```

### Construire et pousser les deux versions

```bash
# Se connecter à Docker Hub
docker login
# Entrer ton username et mot de passe Docker Hub

# ── Construire la version v1 ─────────────────────────────────────
docker build -t TONUSERNAME/iot-playground:v1 .
# -t = tag (NOM:VERSION)
# .  = contexte de build (dossier courant)

# Tester v1 localement
docker run -p 8888:8888 TONUSERNAME/iot-playground:v1
# Dans un autre terminal :
curl http://localhost:8888/
# Attendu : {"status":"ok","message":"v1",...}

# Arrêter le conteneur de test
docker ps                          # Voir les conteneurs en cours
docker stop NOM_OU_ID_CONTENEUR    # Arrêter proprement
# ou Ctrl+C si en premier plan

# ── Construire la version v2 ─────────────────────────────────────
# Méthode 1 : --build-arg (passe une valeur au ARG du Dockerfile)
docker build --build-arg VERSION=v2 -t TONUSERNAME/iot-playground:v2 .

# Méthode 2 : --env au runtime (ne change pas l'image, juste la variable d'env)
docker run -p 8888:8888 -e APP_VERSION=v2 TONUSERNAME/iot-playground:v1
# → Affiche v2 mais c'est toujours l'image v1 (moins propre)

# ── Pousser sur Docker Hub ────────────────────────────────────────
docker push TONUSERNAME/iot-playground:v1
docker push TONUSERNAME/iot-playground:v2

# Vérifier sur Docker Hub
# https://hub.docker.com/r/TONUSERNAME/iot-playground
# → Tu dois voir les deux tags v1 et v2
```

## 10.1 K3d — Kubernetes dans Docker

### Pourquoi K3d pour la Partie 3 ?

Dans les Parties 1 et 2, on a créé des VMs avec Vagrant pour héberger K3s. C'est bien pour comprendre la virtualisation, mais c'est lent (5-10 min pour créer les VMs) et lourd (500 Mo+ par VM).

La Partie 3 utilise **K3d** : c'est K3s qui tourne dans des **conteneurs Docker**. Chaque "nœud" du cluster est un conteneur Docker.

```
PARTIE 1 & 2 (K3s sur VMs Vagrant) :
  Ta machine → VirtualBox → VM → K3s
  Démarrage : 5-10 minutes
  RAM : 512 Mo - 1 Go par VM

PARTIE 3 (K3d = K3s dans Docker) :
  Ta machine → Docker → Conteneur K3s
  Démarrage : 30 secondes
  RAM : ~200 Mo par nœud
```

### Comment K3d fonctionne

```
┌────────────────────────────────────────────────────────────────────┐
│                       TA MACHINE                                    │
│                                                                    │
│   ┌────────────────────────────────────────────────────────────┐   │
│   │                    DOCKER ENGINE                            │   │
│   │                                                            │   │
│   │  ┌──────────────────┐    ┌──────────────────┐              │   │
│   │  │ Conteneur Docker │    │ Conteneur Docker │              │   │
│   │  │                  │    │                  │              │   │
│   │  │  K3S SERVER      │    │  K3S AGENT       │              │   │
│   │  │  (control plane) │    │  (worker node)   │              │   │
│   │  │                  │    │                  │              │   │
│   │  │  [API Server]    │    │  [kubelet]       │              │   │
│   │  │  [Scheduler]     │    │  [kube-proxy]    │              │   │
```

### Installer K3d

```bash
# Vérification
```

### Créer et gérer un cluster K3d

```bash
# ── Créer un cluster ─────────────────────────────────────────────
k3d cluster create mycluster
# Crée un cluster simple avec 1 server + 1 agent

# Avec des options utiles pour ce projet :
k3d cluster create mycluster \
    --port "8080:80@loadbalancer" \
    --port "8443:443@loadbalancer" \
    --wait
#
# --port "8080:80@loadbalancer" :
#   "8080" = port sur ta machine hôte
#   "80"   = port dans le cluster
#   "@loadbalancer" = sur le conteneur load balancer de K3d
#   → localhost:8080 accède au port 80 du cluster
#
# --wait : attendre que tous les nœuds soient Ready avant de rendre la main

# ── Lister les clusters ──────────────────────────────────────────
k3d cluster list
# NAME        SERVERS   AGENTS   LOADBALANCER
# mycluster   1/1       1/1      true

# ── Voir les nœuds ───────────────────────────────────────────────
kubectl get nodes
# NAME                     STATUS   ROLES                  AGE   VERSION
# k3d-mycluster-server-0   Ready    control-plane,master   1m    v1.27.x
# k3d-mycluster-agent-0    Ready    <none>                 1m    v1.27.x

# ── Arrêter / démarrer ───────────────────────────────────────────
k3d cluster stop mycluster        # Arrête les conteneurs (état conservé)
k3d cluster start mycluster       # Redémarre

# ── Supprimer un cluster ─────────────────────────────────────────
k3d cluster delete mycluster      # Supprime tout (irréversible !)

# ── Infos sur le kubeconfig ──────────────────────────────────────
# K3d met automatiquement à jour ~/.kube/config
kubectl config get-contexts
# CURRENT   NAME            CLUSTER         AUTHINFO
# *         k3d-mycluster   k3d-mycluster   admin@k3d-mycluster
```

## 10.2 GitOps — Philosophie et fonctionnement

### Qu'est-ce que le GitOps ?

GitOps est une **façon de gérer l'infrastructure** où :
- Git est la **seule source de vérité**
- Tout changement d'infrastructure passe par un **commit Git**
- Un agent automatique (Argo CD) **synchronise** le cluster avec Git

```
APPROCHE TRADITIONNELLE (imperative) :
  Ingénieur → kubectl apply → Cluster
  Problème : Qui a changé quoi ? Quand ? Pourquoi ?
             Impossible à reproduire exactement
             Pas de rollback facile

APPROCHE GITOPS (declarative) :
  Ingénieur → git push → GitHub → Argo CD → Cluster
  Avantages :
  ✓ Historique complet dans Git (qui, quoi, quand, pourquoi)
  ✓ Rollback = git revert (revenir en arrière en 30 secondes)
  ✓ Reproductible exactement (même repo = même cluster)
  ✓ Collaboration via Pull Requests (revue de code)
  ✓ Audit automatique (toute modification est tracée)
```

### Le cycle GitOps en détail

```
┌──────────────────────────────────────────────────────────────────────┐
│                        CYCLE GITOPS                                   │
│                                                                      │
│   1. COMMIT                                                          │
│   Développeur modifie deployment.yaml :                              │
│   image: wil42/playground:v1  →  image: wil42/playground:v2         │
│   git commit -m "Update to v2" && git push                           │
│                         │                                            │
│                         ▼                                            │
│   2. DÉTECTION                                                       │
│   Argo CD poll GitHub toutes les 3 minutes                           │
│   "Le commit HEAD a changé ! deployment.yaml est différent"          │
│                         │                                            │
│                         ▼                                            │
│   3. COMPARAISON                                                     │
│   Argo CD compare :                                                  │
│   État Git (désiré) : image: wil42/playground:v2                    │
│   État K8s (réel)   : image: wil42/playground:v1                    │
│   → Différence détectée → État = OutOfSync                           │
│                         │                                            │
│                         ▼                                            │
│   4. SYNCHRONISATION (si syncPolicy.automated)                      │
│   Argo CD applique les changements dans K8s :                        │
│   kubectl apply -f deployment.yaml                                   │
│   → K8s lance le rolling update                                      │
│   → Ancien pod supprimé, nouveau pod avec v2 créé                   │
│                         │                                            │
│                         ▼                                            │
```

## 10.3 Argo CD — Cours complet

### Qu'est-ce qu'Argo CD ?

Argo CD est un **outil de déploiement continu GitOps** pour Kubernetes. Il surveille un dépôt Git et synchronise automatiquement le cluster Kubernetes avec le contenu de ce dépôt.

### L'architecture d'Argo CD

```
┌─────────────────────────────────────────────────────────────────┐
│                 CLUSTER KUBERNETES                               │
│                                                                 │
│  Namespace: argocd                                              │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                                                           │  │
│  │  ┌─────────────────┐   Rôle :                             │  │
│  │  │  argocd-server  │ → Interface web (port 443)           │  │
│  │  │                 │   CLI argocd                          │  │
│  │  │                 │   API REST                            │  │
│  │  └─────────────────┘                                      │  │
│  │                                                           │  │
│  │  ┌──────────────────────┐   Rôle :                        │  │
│  │  │  argocd-repo-server  │ → Clone les dépôts Git          │  │
│  │  │                      │   Parse les manifestes YAML     │  │
│  │  │                      │   Calcule les différences       │  │
│  │  └──────────────────────┘                                 │  │
│  │                                                           │  │
│  │  ┌──────────────────────────────────┐   Rôle :            │  │
│  │  │  argocd-application-controller   │ → Compare Git vs K8s│  │
│  │  │                                  │   Lance les syncs   │  │
│  │  │                                  │   Surveille la santé│  │
│  │  └──────────────────────────────────┘                     │  │
│  │                                                           │  │
│  │  ┌──────────────────┐   Rôle :                            │  │

│  │  └──────────────────┘                                     │  │
```

### L'objet Application Argo CD — Expliqué mot par mot

```yaml
apiVersion: argoproj.io/v1alpha1
# ↑ "argoproj.io" = groupe API d'Argo CD
#   "v1alpha1" = version (alpha = encore en évolution, mais stable en pratique)

kind: Application
# ↑ Type d'objet : une "Application" Argo CD
#   C'est un CRD (Custom Resource Definition) = un type K8s ajouté par Argo CD

metadata:
  name: wil-playground
  # ↑ Nom de l'application dans Argo CD
  #   Visible dans l'interface web et la CLI
  
  namespace: argocd
  # ↑ L'objet Application DOIT être dans le namespace argocd
  #   Argo CD surveille seulement les Applications dans son propre namespace

spec:
  project: default
  # ↑ Projet Argo CD (pour organiser les applications)
  #   "default" = projet par défaut, pas de restrictions
  #   Les projets permettent de contrôler qui peut déployer quoi

  source:
  # ↑ Où Argo CD va chercher les manifestes à déployer
  
    repoURL: https://github.com/TONUSERNAME/TONREPO.git
    # ↑ URL complète du dépôt Git
    #   DOIT être public OU avoir des credentials configurés
    #   Format HTTPS (pas SSH) pour les dépôts publics
    
    targetRevision: HEAD
    # ↑ Quelle version du dépôt surveiller
    #   HEAD = toujours la dernière version de la branche principale
    #   On peut aussi mettre :
    #   - "main" ou "master" (nom de branche)
    #   - "v1.2.3" (tag Git)
    #   - "abc123def" (hash de commit spécifique)
    
    path: p3/confs
    # ↑ Dossier dans le dépôt qui contient les fichiers YAML à déployer
    #   Argo CD va appliquer TOUS les fichiers .yaml dans ce dossier
    #   Structure attendue :
    #   p3/confs/
    #   ├── deployment.yaml   ← sera appliqué
    #   └── service.yaml      ← sera appliqué (si tu le crées séparément)

  destination:
  # ↑ Où déployer dans le cluster Kubernetes
  
    server: https://kubernetes.default.svc
    # ↑ L'API Server cible
    #   "https://kubernetes.default.svc" = le cluster LOCAL (celui où Argo CD tourne)
    #   On peut aussi pointer vers des clusters distants !
    
    namespace: dev
    # ↑ Dans quel namespace déployer les ressources
    #   Les Deployments, Services, etc. seront créés dans ce namespace

  syncPolicy:
  # ↑ Comment Argo CD gère la synchronisation
  
    automated:
    # ↑ Synchronisation AUTOMATIQUE (sans intervention humaine)
    #   Sans "automated", il faut cliquer "Sync" dans l'interface
    
      prune: true
      # ↑ Si un fichier est SUPPRIMÉ du dépôt Git,
      #   Argo CD SUPPRIME aussi la ressource correspondante dans K8s
      #   false = les ressources restent même si supprimées de Git
      
      selfHeal: true
      # ↑ Si quelqu'un modifie manuellement une ressource dans K8s
      #   (ex: kubectl edit deployment), Argo CD la RESTAURE à l'état Git
      #   C'est le cœur du GitOps : Git = source de vérité unique
      #   false = les modifications manuelles ne sont pas corrigées
    
    syncOptions:
    - CreateNamespace=true
    # ↑ Si le namespace "dev" n'existe pas, Argo CD le CRÉE automatiquement
    #   Sans ça, la sync échouerait si le namespace est absent
    
    retry:
    # ↑ En cas d'échec de synchronisation, réessayer automatiquement
      limit: 5
      # ↑ Maximum 5 tentatives
      backoff:
        duration: 5s          # Attendre 5s avant la 1ère retry
        maxDuration: 3m       # Jamais attendre plus de 3 minutes entre retries
        factor: 2             # Doubler le temps d'attente à chaque retry
        # → 5s, 10s, 20s, 40s, 80s (max 3m)
```

## 11.3 Script d'installation complet

```bash
nano p3/scripts/install.sh
```

```bash
#!/bin/bash
# ════════════════════════════════════════════════════════════════════
# SCRIPT D'INSTALLATION COMPLET — PARTIE 3
# Usage : bash p3/scripts/install.sh
#
# Ce script installe tout depuis zéro :
# 1. Docker
# 2. kubectl
# 3. K3d
# 4. Cluster K3d
# 5. Namespaces (argocd + dev)
# 6. Argo CD
# 7. Configuration de l'application
# ════════════════════════════════════════════════════════════════════

# set -e : arrête le script à la première erreur
# set -u : erreur si variable non définie
# set -o pipefail : erreurs dans les pipes remontent
set -euo pipefail

# ── Couleurs pour l'affichage ─────────────────────────────────────
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
BOLD='\033[1m'
NC='\033[0m'  # No Color (reset)

# ── Fonctions d'affichage ─────────────────────────────────────────
info()    { echo -e "${BLUE}[INFO]${NC}    $1"; }
success() { echo -e "${GREEN}[OK]${NC}      $1"; }
warning() { echo -e "${YELLOW}[WARN]${NC}   $1"; }
error()   { echo -e "${RED}[ERREUR]${NC} $1"; exit 1; }
step()    { echo -e "\n${BOLD}══ $1 ══${NC}"; }

# ── Vérification du système d'exploitation ───────────────────────
if [[ "$OSTYPE" != "linux-gnu"* ]]; then
    warning "Ce script est prévu pour Linux. OS détecté : $OSTYPE"
fi

echo ""
echo -e "${BOLD}╔══════════════════════════════════════════════╗${NC}"
echo -e "${BOLD}║   Installation IoT — Partie 3                ║${NC}"
echo -e "${BOLD}║   K3d + Argo CD + GitOps                     ║${NC}"
echo -e "${BOLD}╚══════════════════════════════════════════════╝${NC}"
echo ""

# ════════════════════════════════════════════════════════════════════
# ÉTAPE 1 : DOCKER
# ════════════════════════════════════════════════════════════════════
step "Étape 1/5 : Docker"

if command -v docker &>/dev/null; then
    success "Docker déjà installé : $(docker --version)"
else
    info "Installation de Docker..."
    curl -fsSL https://get.docker.com | sh
    sudo usermod -aG docker "$USER"
    success "Docker installé !"
    warning "Ferme et rouvre ton terminal pour que Docker soit accessible sans sudo"
    warning "Ou exécute : newgrp docker"
fi

# Vérifier que Docker tourne
if ! docker info &>/dev/null; then
    info "Démarrage du service Docker..."
    sudo systemctl start docker
    sudo systemctl enable docker
fi
success "Docker opérationnel"

# ════════════════════════════════════════════════════════════════════
# ÉTAPE 2 : KUBECTL
# ════════════════════════════════════════════════════════════════════
step "Étape 2/5 : kubectl"

if command -v kubectl &>/dev/null; then
    success "kubectl déjà installé : $(kubectl version --client --short 2>/dev/null || true)"
else
    info "Téléchargement de kubectl..."
    KUBECTL_VER=$(curl -L -s https://dl.k8s.io/release/stable.txt)
    info "Version : $KUBECTL_VER"
    curl -LO "https://dl.k8s.io/release/${KUBECTL_VER}/bin/linux/amd64/kubectl"
    chmod +x kubectl
    sudo mv kubectl /usr/local/bin/
    success "kubectl installé : $(kubectl version --client --short 2>/dev/null)"
fi

# ════════════════════════════════════════════════════════════════════
# ÉTAPE 3 : K3D
# ════════════════════════════════════════════════════════════════════
step "Étape 3/5 : K3d"

if command -v k3d &>/dev/null; then
    success "K3d déjà installé : $(k3d version | head -1)"
else
    info "Installation de K3d..."
    curl -s https://raw.githubusercontent.com/k3d-io/k3d/main/install.sh | bash
    success "K3d installé : $(k3d version | head -1)"
fi

# ════════════════════════════════════════════════════════════════════
# ÉTAPE 4 : CLUSTER K3D
# ════════════════════════════════════════════════════════════════════
step "Étape 4/5 : Cluster K3d"

CLUSTER_NAME="mycluster"

if k3d cluster list 2>/dev/null | grep -q "^${CLUSTER_NAME}"; then
    warning "Le cluster '${CLUSTER_NAME}' existe déjà"
    info "Vérification qu'il est démarré..."
    k3d cluster start "${CLUSTER_NAME}" 2>/dev/null || true
else
    info "Création du cluster K3d '${CLUSTER_NAME}'..."
    k3d cluster create "${CLUSTER_NAME}" \
        --port "8080:80@loadbalancer" \
        --port "8443:443@loadbalancer" \
        --wait
    success "Cluster '${CLUSTER_NAME}' créé !"
fi

# Attendre que les nœuds soient prêts
info "Attente que les nœuds soient Ready..."
kubectl wait --for=condition=Ready nodes --all --timeout=120s
success "Cluster opérationnel !"
echo ""
kubectl get nodes
echo ""

# ════════════════════════════════════════════════════════════════════
# ÉTAPE 5 : NAMESPACES + ARGO CD
# ════════════════════════════════════════════════════════════════════
step "Étape 5/5 : Namespaces + Argo CD"

# Créer les namespaces
# "--dry-run=client -o yaml | kubectl apply -f -" = idempotent
# (peut être exécuté plusieurs fois sans erreur même si le namespace existe)
info "Création des namespaces..."
kubectl create namespace argocd --dry-run=client -o yaml | kubectl apply -f -
kubectl create namespace dev    --dry-run=client -o yaml | kubectl apply -f -
success "Namespaces créés :"
kubectl get namespaces | grep -E "argocd|dev"

# Installer Argo CD
info "Installation d'Argo CD..."
kubectl apply -n argocd \
    -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Attendre qu'Argo CD soit opérationnel
info "Attente qu'Argo CD soit prêt (peut prendre 2-4 minutes)..."
info "C'est normal, il télécharge plusieurs images Docker..."

# Attendre le déploiement principal
kubectl wait --for=condition=Available \
    deployment/argocd-server \
    -n argocd \
    --timeout=300s

success "Argo CD est prêt !"

# Appliquer la configuration de l'application
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
CONFS_DIR="${SCRIPT_DIR}/../confs"
ARGOCD_APP="${CONFS_DIR}/argocd-app.yaml"

if [ -f "${ARGOCD_APP}" ]; then
    info "Application de la configuration Argo CD..."
    kubectl apply -f "${ARGOCD_APP}"
    success "Application Argo CD configurée !"
else
    warning "Fichier argocd-app.yaml non trouvé dans : ${CONFS_DIR}"
    warning "Applique-le manuellement après avoir mis à jour l'URL du repo :"
    warning "  kubectl apply -f p3/confs/argocd-app.yaml"
fi

# ════════════════════════════════════════════════════════════════════
# RÉSUMÉ FINAL
# ════════════════════════════════════════════════════════════════════

# Récupérer le mot de passe Argo CD
ARGOCD_PASS=$(kubectl -n argocd get secret argocd-initial-admin-secret \
    -o jsonpath="{.data.password}" 2>/dev/null | base64 -d 2>/dev/null || echo "non disponible")

echo ""
echo -e "${BOLD}╔══════════════════════════════════════════════════════════════╗${NC}"
echo -e "${BOLD}║                   INSTALLATION TERMINÉE !                    ║${NC}"
echo -e "${BOLD}╚══════════════════════════════════════════════════════════════╝${NC}"
echo ""
echo -e "${GREEN}✓${NC} Cluster K3d : ${CLUSTER_NAME}"
echo -e "${GREEN}✓${NC} Namespaces  : argocd, dev"
echo -e "${GREEN}✓${NC} Argo CD     : opérationnel"
echo ""
echo -e "${BOLD}─── Accès à l'interface Argo CD ───────────────────────────────${NC}"
echo "  Commande : kubectl port-forward svc/argocd-server -n argocd 8443:443 &"
echo "  URL      : https://localhost:8443  (accepte le certificat auto-signé)"
echo "  User     : admin"
echo "  Password : ${ARGOCD_PASS}"
echo ""
echo -e "${BOLD}─── Accès à l'application ─────────────────────────────────────${NC}"
echo "  Commande : kubectl port-forward svc/wil-playground -n dev 8888:8888 &"
echo "  Test     : curl http://localhost:8888/"
echo "  Attendu  : {\"status\":\"ok\",\"message\":\"v1\"}"
echo ""
echo -e "${BOLD}─── Démonstration de la mise à jour automatique ───────────────${NC}"
echo "  1. Modifier p3/confs/deployment.yaml : v1 → v2"
echo "  2. git add . && git commit -m 'v2' && git push"
echo "  3. Attendre ~3 minutes (ou forcer dans l'UI Argo CD)"
echo "  4. curl http://localhost:8888/ → {\"message\": \"v2\"}"
echo ""
```

### Installer GitLab (configuration minimale)

> ⚠️ **GitLab est TRÈS gourmand en ressources !**
> - RAM minimum recommandée : **8 Go libres**
> - Temps d'installation : **15-30 minutes**
> - Si ton système a moins de RAM, GitLab sera très lent ou instable

```bash
# Installation de GitLab Community Edition (CE = gratuit)

  --namespace gitlab \
  --timeout 600s \
  --set global.edition=ce \
  --set global.hosts.domain=localhost \
  --set global.hosts.https=false \
  --set global.ingress.configureCertmanager=false \
  --set global.ingress.enabled=false \
  --set certmanager.install=false \
  --set certmanager-issuer.email=admin@localhost \
  --set nginx-ingress.enabled=false \
  --set prometheus.install=false \
  --set grafana.enabled=false \
  --set gitlab-runner.install=false \
  --set registry.enabled=false \
  --set minio.persistence.size=2Gi \
  --set postgresql.persistence.size=2Gi

# Explication des options :
# global.edition=ce         → Community Edition (gratuit)
# global.hosts.domain=localhost → Accès en local
# certmanager.install=false  → Pas de gestion automatique des certificats
# nginx-ingress.enabled=false → On utilise l'Ingress de K3d (Traefik)
# prometheus.install=false   → Pas de monitoring (économise la RAM)
# gitlab-runner.install=false → Pas de CI/CD automatique
# registry.enabled=false     → Pas de registre Docker intégré
```

# CHAPITRE 13 — DÉPANNAGE COMPLET

## 13.1 Méthodologie : comment déboguer Kubernetes

Face à un problème, suis toujours cet ordre :

```
ÉTAPE 1 : Identifier QUEL objet a un problème
  kubectl get pods -A    → Cherche les états non-Running
  kubectl get nodes      → Vérifie que les nœuds sont Ready

ÉTAPE 2 : Comprendre POURQUOI
  kubectl describe pod NOM_DU_POD
  → Lis la section "Events" (tout en bas) — c'est la clef !

ÉTAPE 3 : Lire les logs de l'application
  kubectl logs NOM_DU_POD
  → Qu'est-ce que l'application dit elle-même ?

ÉTAPE 4 : Tester le réseau
  kubectl exec -it NOM_DU_POD -- wget -O- http://autre-service
  → Est-ce que les pods peuvent se parler ?

ÉTAPE 5 : Corriger et relancer
  kubectl delete pod NOM_DU_POD   → K8s en recrée un propre
  kubectl rollout restart deployment/NOM  → Redémarre proprement
```

---

## 13.2 Les états des pods et ce qu'ils signifient

| État | Signification | Action |
|------|--------------|--------|
| `Pending` | Le pod attend d'être schedulé | `kubectl describe pod` → Events |
| `ContainerCreating` | Le pod démarre, télécharge l'image | Attendre (normal) |
| `Running` | Le pod fonctionne | ✅ Tout va bien |
| `CrashLoopBackOff` | Le conteneur plante en boucle | `kubectl logs pod --previous` |
| `ImagePullBackOff` | Impossible de télécharger l'image | Vérifier le nom et l'accès internet |
| `OOMKilled` | Tué car Out Of Memory | Augmenter les limites mémoire |
| `Terminating` | En cours de suppression | Attendre (normal) |
| `Evicted` | Supprimé par manque de ressources | Libérer de la RAM/disque |
| `Error` | Erreur au démarrage | `kubectl logs pod` |

```bash
k3d cluster create mycluster --port "9080:80@loadbalancer"

# ─── Cluster K3d ne démarre pas ──────────────────────────────────
# Supprimer et recréer :
k3d cluster delete mycluster
k3d cluster create mycluster --port "8080:80@loadbalancer" --wait

# ─── kubectl ne voit pas le cluster K3d ──────────────────────────
kubectl config get-contexts
# Si k3d-mycluster n'est pas là :
k3d kubeconfig get mycluster > ~/.kube/config
# ou
kubectl config use-context k3d-mycluster

# ─── Les pods K3d ont des erreurs réseau ─────────────────────────
# Parfois les conteneurs K3d ont des problèmes de réseau après un sleep/wake
docker restart k3d-mycluster-server-0
docker restart k3d-mycluster-agent-0
docker restart k3d-mycluster-serverlb
```

# CHAPITRE 14 — GLOSSAIRE EXHAUSTIF

## A

**Annotation**
Paire clé-valeur attachée à un objet Kubernetes pour stocker des métadonnées non utilisées pour la sélection. Contrairement aux labels, les annotations ne sont pas filtrables. Utilisées pour les informations de déploiement, les configs d'outils (Argo CD, Traefik), etc.
```yaml
annotations:
  kubectl.kubernetes.io/last-applied-configuration: ...
  kubernetes.io/ingress.class: "traefik"
```

**API Server (kube-apiserver)**
Composant central du Control Plane de Kubernetes. Toutes les communications (kubectl, kubelet, Argo CD...) passent par lui. Expose une API REST sur le port 6443 (HTTPS).

**APK**
Gestionnaire de paquets d'Alpine Linux. Équivalent de `apt` pour Ubuntu/Debian.
```bash
apk add nginx    # Installer
apk del nginx    # Désinstaller
apk update       # Mettre à jour la liste
```

**Argo CD**
Outil de déploiement continu GitOps pour Kubernetes. Surveille un dépôt Git et synchronise automatiquement le cluster avec le contenu de ce dépôt.

---

## B

**Base image**
Image Docker sur laquelle on se base pour construire la nôtre (instruction `FROM` dans le Dockerfile).

**Box (Vagrant)**
Image de machine virtuelle pré-construite utilisée par Vagrant. Hébergée sur https://app.vagrantup.com.

**Build**
Action de construire une image Docker depuis un Dockerfile.
```bash
docker build -t mon-image:v1 .
```

---

## C

**cgroups (Control Groups)**
Mécanisme du noyau Linux qui limite et surveille les ressources (CPU, RAM, réseau, disque) d'un groupe de processus. C'est ce qui permet à Docker de garantir qu'un conteneur ne dépasse pas sa limite mémoire.

**Chart (Helm)**
Package Helm contenant des templates de fichiers YAML Kubernetes et des valeurs par défaut. Permet d'installer des applications complexes en une commande.

**CI/CD**
- **CI** (Continuous Integration) : Tests automatiques à chaque commit Git
- **CD** (Continuous Delivery/Deployment) : Déploiement automatique après CI réussi. Argo CD gère le CD dans ce projet.

**Cluster**
Ensemble de machines (nodes) qui travaillent ensemble sous la gestion de Kubernetes.

**ClusterIP**
Type de Service Kubernetes. IP stable accessible uniquement depuis l'intérieur du cluster. Utilisée pour la communication inter-pods.

**ConfigMap**
Objet Kubernetes qui stocke des données de configuration non-sensibles (chaînes de caractères, fichiers de config). Injecté dans les pods via variables d'environnement ou volumes.
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: ma-config
data:
  DATABASE_URL: "mysql:3306"
  DEBUG: "false"
```

**containerd**
Runtime de conteneurs (container runtime) utilisé par K3s et la majorité des installations Kubernetes modernes. C'est lui qui télécharge les images et lance réellement les conteneurs.

**Conteneur**
Instance en cours d'exécution d'une image Docker. Processus isolé grâce aux namespaces et cgroups Linux.

**Controller Manager (kube-controller-manager)**
Composant du Control Plane qui exécute des boucles de contrôle pour surveiller l'état du cluster et corriger les écarts avec l'état désiré.

**CoreDNS**
Serveur DNS interne de Kubernetes. Résout les noms de services en IP. Installé automatiquement avec K3s.
```
Service "mon-app" dans namespace "default"
→ DNS : mon-app.default.svc.cluster.local
→ IP  : 10.96.x.x (ClusterIP)
```

**CRD (Custom Resource Definition)**
Mécanisme permettant d'ajouter de nouveaux types d'objets à Kubernetes. Argo CD utilise des CRDs pour ses objets `Application`, `AppProject`, etc.

**CrashLoopBackOff**
État d'un pod qui plante et redémarre en boucle. À diagnostiquer avec `kubectl logs pod --previous`.

---

## D

**Deployment**
Objet Kubernetes qui gère un ensemble de pods identiques. Garantit le nombre de réplicas, effectue les rolling updates, permet les rollbacks.

**Docker**
Plateforme de conteneurisation. Composée du docker CLI, du docker daemon, et de l'écosystème (Docker Hub, Docker Compose, etc.).

**Dockerfile**
Fichier texte contenant les instructions pour construire une image Docker.

**Docker Hub**
Registre public d'images Docker. Héberge des millions d'images officielles et communautaires.

**Docker Engine / Daemon (dockerd)**
Le serveur Docker qui tourne en arrière-plan et gère les conteneurs, images, réseaux et volumes.

---

## E

**Endpoint**
Association IP:Port d'un pod, maintenue par un Service Kubernetes. Quand un pod est créé avec les bons labels, il est ajouté automatiquement aux endpoints du Service correspondant.
```bash
kubectl get endpoints app-one
# NAME      ENDPOINTS                    AGE
# app-one   10.42.0.15:80,10.42.0.16:80  5m
```

**etcd**
Base de données clé-valeur distribuée utilisée par Kubernetes pour stocker tout l'état du cluster. Remplacé par SQLite dans K3s par défaut.

---

## G

**Git**
Système de contrôle de versions distribué. Garde l'historique complet de tous les fichiers d'un projet.

**GitHub**
Plateforme web d'hébergement de dépôts Git. Utilisée dans ce projet comme source de vérité pour Argo CD.

**GitLab**
Plateforme Git auto-hébergeable (alternative à GitHub). Utilisée dans le bonus du projet.

**GitOps**
Pratique DevOps où Git est la source de vérité unique pour l'état de l'infrastructure. Tout changement passe par un commit Git, et un agent (Argo CD) applique automatiquement les changements au cluster.

---

## H

**Health Check (Probe)**
Mécanisme Kubernetes pour vérifier l'état d'un conteneur.
- **Liveness probe** : si échec → redémarre le conteneur
- **Readiness probe** : si échec → retire du Service (plus de trafic)
- **Startup probe** : vérifie que l'application a bien démarré

**Helm**
Gestionnaire de packages pour Kubernetes. Permet d'installer des applications complexes avec une seule commande via des "charts".

**Host (HTTP)**
En-tête HTTP envoyé par le navigateur pour indiquer quel site il demande. Utilisé par l'Ingress pour router vers le bon Service.
```
curl -H "Host: app1.com" http://192.168.56.110
→ L'Ingress route vers app-one grâce à cet en-tête
```

**Hyperviseur**
Logiciel qui crée et gère des machines virtuelles. VirtualBox est un hyperviseur de type 2 (tourne sur un OS hôte).

---

## I

**Image Docker**
Template immuable (lecture seule) contenant tout le nécessaire pour exécuter une application : code, runtime, bibliothèques, variables d'environnement.

**ImagePullBackOff**
État d'un pod qui ne peut pas télécharger son image Docker. Causes : image inexistante, typo dans le nom, image privée sans credentials, pas d'accès internet.

**Ingress**
Objet Kubernetes qui définit des règles de routage HTTP/HTTPS. Redirige le trafic entrant vers les bons Services selon le nom de domaine ou le chemin URL.

**Ingress Controller**
Logiciel qui implémente les règles Ingress. K3s utilise **Traefik** par défaut. Sans Ingress Controller, les objets Ingress ne fonctionnent pas.

**IP (Internet Protocol)**
Adresse numérique identifiant une machine sur un réseau. Format IPv4 : 4 nombres de 0 à 255 séparés par des points (ex: 192.168.56.110).

---

## K

**K3d**
Outil qui fait tourner K3s dans des conteneurs Docker. Permet de créer des clusters Kubernetes en quelques secondes.

**K3s**
Distribution légère de Kubernetes créée par Rancher Labs. Idéale pour l'apprentissage, l'IoT, et les environnements aux ressources limitées.

**K8s**
Abréviation de "Kubernetes" (K + 8 lettres + s).

**kubectl**
Outil en ligne de commande pour interagir avec un cluster Kubernetes. Communique avec l'API Server via HTTPS.

**kubelet**
Agent Kubernetes qui tourne sur chaque worker node. Reçoit les ordres de l'API Server et lance/surveille les pods.

**kubeconfig**
Fichier de configuration pour kubectl (par défaut `~/.kube/config`). Contient les informations de connexion au(x) cluster(s).

**Kubernetes**
Système open-source d'orchestration de conteneurs, créé par Google en 2014. Automatise le déploiement, la mise à l'échelle et la gestion des applications conteneurisées.

---

## L

**Label**
Paire clé-valeur attachée à un objet Kubernetes. Utilisée pour identifier, organiser et sélectionner des ressources.
```yaml
labels:
  app: nginx
  env: production
  version: "1.2"
```

**LoadBalancer**
Type de Service Kubernetes qui crée un équilibreur de charge cloud (AWS ELB, GCP LB...). Attribue une IP publique au Service.

**Liveness Probe**
Vérification périodique de la santé d'un conteneur. Si elle échoue plusieurs fois → K8s redémarre le conteneur.

---

## M

**Manifest**
Fichier YAML décrivant un objet Kubernetes. "Appliquer un manifest" = `kubectl apply -f fichier.yaml`.

**Microservices**
Architecture logicielle où une application est décomposée en petits services indépendants. Kubernetes est conçu pour gérer des architectures microservices.

---

## N

**Namespace**
Partition logique d'un cluster Kubernetes. Permet d'isoler les ressources par équipe, environnement ou application.

**NAT (Network Address Translation)**
Mécanisme réseau qui traduit les adresses IP privées en adresses publiques. Utilisé par Vagrant pour donner l'accès internet aux VMs.

**Node (Nœud)**
Machine physique ou virtuelle dans un cluster Kubernetes. Peut être un master/control plane ou un worker.

**NodePort**
Type de Service Kubernetes qui expose le service sur un port de chaque nœud (30000-32767). Accessible depuis l'extérieur du cluster.

---

## O

**OOMKilled**
État d'un pod tué par le noyau Linux car il a dépassé sa limite mémoire (Out Of Memory). Solution : augmenter `resources.limits.memory`.

**Orchestration**
Gestion automatisée du déploiement, de la mise à l'échelle, du réseau et de la disponibilité des applications conteneurisées. C'est le rôle principal de Kubernetes.

---

## P

**Pending**
État d'un pod qui attend d'être schedulé. Causes typiques : ressources insuffisantes, image en cours de téléchargement, nœud non disponible.

**Pod**
Plus petite unité déployable dans Kubernetes. Contient un ou plusieurs conteneurs qui partagent le réseau et le stockage. Les pods sont éphémères — si un pod est supprimé, un nouveau pod (avec une nouvelle IP) est créé.

**Port-forward**
Commande kubectl qui redirige un port local vers un port dans le cluster. Utilisé pour tester des applications sans les exposer publiquement.
```bash
kubectl port-forward svc/mon-service 8080:80
# localhost:8080 → service:80 dans le cluster
```

**Private Network (Vagrant)**
Type de réseau Vagrant qui crée un réseau privé entre les VMs et la machine hôte. Chaque VM reçoit une IP fixe.

**Provisioner (Vagrant)**
Script ou outil exécuté automatiquement sur une VM après sa création. Dans ce projet : scripts shell qui installent K3s.

---

## R

**Readiness Probe**
Vérification qui détermine si un pod est prêt à recevoir du trafic. Si elle échoue → le pod est retiré du Service (mais pas redémarré).

**Registre (Registry)**
Serveur qui stocke et distribue des images Docker. Docker Hub est le registre public principal.

**ReplicaSet**
Objet Kubernetes qui garantit qu'un nombre spécifié de pods identiques tournent en permanence. Créé et géré automatiquement par un Deployment.

**Rolling Update**
Stratégie de mise à jour Kubernetes qui remplace les pods un par un, garantissant zéro downtime.

**Rollback**
Retour à une version précédente d'un déploiement.
```bash
kubectl rollout undo deployment/mon-app         # Revenir à la version précédente
kubectl rollout undo deployment/mon-app --to-revision=3  # Revenir à la révision 3
```

---

## S

**Scheduler (kube-scheduler)**
Composant du Control Plane qui décide sur quel nœud placer un nouveau pod, en fonction des ressources disponibles et des contraintes.

**Secret**
Objet Kubernetes qui stocke des données sensibles (mots de passe, tokens, certificats) encodées en base64. Les secrets peuvent être injectés dans les pods via variables d'environnement ou volumes.

**Selector**
Requête qui filtre les objets Kubernetes par leurs labels. Utilisé par les Services pour trouver leurs pods, par les Deployments pour gérer leurs ReplicaSets, etc.

**Service**
Abstraction réseau stable qui expose un ensemble de pods. Fournit une IP et un nom DNS stables, indépendants des pods individuels (qui peuvent changer).

**SSH (Secure Shell)**
Protocole de communication sécurisé pour accéder à distance à une machine Linux en ligne de commande.

**Sidecar**
Conteneur auxiliaire dans un pod, qui assiste le conteneur principal (ex: collecteur de logs, proxy, agent de monitoring).

**syncPolicy (Argo CD)**
Configuration qui détermine comment et quand Argo CD synchronise le cluster avec le dépôt Git. `automated` = synchronisation automatique sans intervention humaine.

---

## T

**Tag (Docker)**
Version ou variante d'une image Docker. Format : `image:tag` (ex: `nginx:alpine`, `wil42/playground:v1`).

**Tagging (Git)**
Marquer un commit spécifique avec un identifiant lisible (ex: `v1.0`, `v2.0`). Utilisé pour identifier les versions d'une application.

**Token (K3s)**
Clé d'authentification partagée entre le serveur K3s et ses agents. Permet aux workers de rejoindre le cluster de façon sécurisée.

**Traefik**
Reverse proxy et Ingress Controller moderne. Inclus par défaut dans K3s. Gère automatiquement le routage HTTP basé sur les règles Ingress.

---

## U

**Union File System**
Système de fichiers en couches utilisé par Docker. Permet de partager les couches communes entre plusieurs images, économisant l'espace disque.

---

## V

**Vagrant**
Outil de gestion de machines virtuelles automatisé. Crée et configure des VMs à partir d'un fichier `Vagrantfile`.

**Vagrantfile**
Fichier de configuration Vagrant écrit en Ruby. Décrit les VMs à créer (box, réseau, ressources, scripts de provisioning).

**VM (Machine Virtuelle)**
Émulation logicielle d'un ordinateur complet. Fonctionne à l'intérieur d'un hyperviseur (VirtualBox) comme si c'était une vraie machine physique.

**Volume**
Espace de stockage persistant pour les pods Kubernetes. Contrairement au système de fichiers d'un conteneur (éphémère), un volume survit au redémarrage d'un pod.

**VirtualBox**
Hyperviseur open-source de type 2 (tourne sur un OS hôte). Utilisé dans ce projet pour héberger les VMs Vagrant.

---

## W

**Worker Node**
Machine dans un cluster Kubernetes dont le rôle est d'exécuter les pods (les applications). Contrôlé par le Control Plane via le kubelet.

---

## Y

**YAML**
Format de sérialisation de données basé sur l'indentation par espaces. Utilisé massivement en Kubernetes pour décrire les objets (Deployments, Services, Ingress...).
Règle absolue : **jamais de tabulations, toujours des espaces !**
